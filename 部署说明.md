# 面试准备助手 GitHub Pages 部署说明

第一版是纯静态 Web MVP，可以直接托管到 GitHub Pages。页面本身不保存用户信息、不保存 API Key、不保存简历、JD 或报告。

## 1. 创建 GitHub 仓库

1. 打开 GitHub 并登录账号。
2. 点击右上角 `+` → `New repository`。
3. `Repository name` 填写仓库名，例如 `offerAgent`。
4. `Description` 可以填写：`Open-source interview preparation MVP powered by user-configured LLM keys`。
5. 选择 `Public`，方便开源展示；如果暂时不想公开，也可以先选 `Private`。
6. 不要勾选 `Add a README file`。
7. 不要选择 `.gitignore` 模板。
8. 不要选择 License 模板。
9. 点击 `Create repository`。

原因：本地项目里已经有 `README.md`、`LICENSE`、`.gitignore`，如果 GitHub 初始化时也创建这些文件，首次推送时容易产生冲突。

创建完成后，GitHub 会显示仓库地址，类似：

```text
https://github.com/<your-name>/<your-repo>.git
```

后续命令里的 `<your-name>` 替换成你的 GitHub 用户名，`<your-repo>` 替换成你的仓库名。

## 2. 推送本地代码

### 2.1 确认当前目录

打开 PowerShell，进入项目目录：

```powershell
cd D:\OfferAgent
```

确认能看到项目文件：

```powershell
dir
```

应该能看到类似：

```text
apps
docs
README.md
LICENSE
部署说明.md
```

### 2.2 检查 Git 状态

```powershell
git status
```

如果提示不是 Git 仓库，先执行：

```powershell
git init
```

当前项目通常已经有 `.git`，所以多数情况下不需要重新初始化。

### 2.3 设置主分支为 master

```powershell
git branch -M master
```

### 2.4 添加远程仓库

首次添加 remote：

```powershell
git remote add origin https://github.com/<your-name>/<your-repo>.git
```

如果提示 `remote origin already exists`，说明之前配置过 remote，改用：

```powershell
git remote set-url origin https://github.com/<your-name>/<your-repo>.git
```

确认 remote：

```powershell
git remote -v
```

应该看到类似：

```text
origin  https://github.com/<your-name>/<your-repo>.git (fetch)
origin  https://github.com/<your-name>/<your-repo>.git (push)
```

### 2.5 提交代码

```powershell
git add .
git commit -m "Initial interview prep web MVP"
```

如果 Git 提示需要配置用户名和邮箱，执行：

```powershell
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
```

然后重新执行：

```powershell
git commit -m "Initial interview prep web MVP"
```

### 2.6 推送到 GitHub

```powershell
git push -u origin master
```

如果 GitHub 要求登录，按提示完成浏览器登录或输入 Token。

完整命令汇总：

```bash
git remote add origin https://github.com/<your-name>/<your-repo>.git
git branch -M master
git add .
git commit -m "Initial interview prep web MVP"
git push -u origin master
```

如果已经配置过 remote，使用这组命令：

```bash
git remote set-url origin https://github.com/<your-name>/<your-repo>.git
git branch -M master
git add .
git commit -m "Initial interview prep web MVP"
git push -u origin master
```

## 3. 启用 GitHub Pages

### 3.1 进入 Pages 设置

1. 打开你的 GitHub 仓库页面。
2. 点击顶部导航里的 `Settings`。
3. 左侧找到 `Pages`。
4. 找到 `Build and deployment`。
5. `Source` 选择 `GitHub Actions`。

### 3.2 等待自动部署

项目已经包含 workflow 文件：

```text
.github/workflows/pages.yml
```

推送到 `master` 后，GitHub 会自动运行部署流程。

查看部署状态：

1. 进入仓库顶部的 `Actions`。
2. 找到 `Deploy static MVP to GitHub Pages`。
3. 等待状态变成绿色对勾。

如果失败，点进去查看错误日志。常见原因见第 8 节。

部署成功后，GitHub 会给出访问地址，通常类似：

```text
https://<your-name>.github.io/<your-repo>/
```

### 3.3 确认访问页面

打开部署地址后，应看到面试准备助手页面，包括：

- 顶部固定标题栏。
- 工作台 / 图谱切换。
- 候选人 / 面试官模式切换。
- 模型服务商与临时模型配置。
- 候选人简历与岗位 JD 输入区。
- Offer 沙盘与面试官视角库。
- 报告预览、证据关系图谱、虚拟委员会和结果摘要。

## 4. 使用方式

1. 打开 GitHub Pages 页面。
2. 选择 `Mock Demo`，或选择模型服务商并临时输入自己的 API Key。主流模型 Key 获取方式见第 6 节。
3. 选择目标岗位，并粘贴简历和 JD。
4. 点击生成报告。
5. 查看评估报告、证据关系图谱、虚拟委员会和结果摘要。
6. 面试官模式下填写人工反馈。
7. 下载与当前角色匹配的 PDF 报告。

### 4.1 本地运行 Mock Demo

本地体验页面时，推荐不要用 IDE 内置预览或直接 `file://` 打开。可以启动一个本地 HTTP 服务：

```powershell
cd D:\OfferAgent\apps\web
python -m http.server 5173
```

然后访问：

```text
http://localhost:5173/
```

如果看到 `Unsafe attempt to load URL file:///...#workspace`，通常是 IDE 预览 iframe 或 `file://` 页面导航被浏览器安全策略拦截，改用 `http://localhost:5173/` 即可。

### 4.2 本地运行真实模型

如果要在本地使用真实模型，推荐启动本地代理：

```powershell
cd D:\OfferAgent
python scripts\local_proxy.py --key-file 1.md
```

启动后访问：

```text
http://127.0.0.1:5173/
```

页面中填写：

- 模型服务商：`OpenAI-Compatible 代理 / 自定义接口`
- 模型名称：`qwen-plus`
- 临时 API Key：你的真实模型 Key
- 代理 / 自定义 Base URL：`http://127.0.0.1:8787`

本地真实模型配置文件 `1.md` 格式如下：

```text
KEY:你的模型 Key
URL:模型服务商 OpenAI-Compatible Base URL
```

例如阿里百炼 / MaaS：

```text
URL:https://ws-ppprfnb8819dlxt6.cn-beijing.maas.aliyuncs.com/compatible-mode/v1
```

`1.md` 已加入 `.gitignore`，不要提交到 GitHub。

### 4.3 功能自测

推送或部署前，可以在本地运行静态功能检查：

```powershell
python scripts\smoke_test.py
```

如果需要测试真实模型流式接口，可在本地创建不提交的 `1.md`：

```text
KEY:你的模型 Key
URL:模型服务商 OpenAI-Compatible Base URL
```

然后执行：

```powershell
python scripts\smoke_test.py --with-llm 1.md --model qwen-plus
```

测试会验证目标岗位选择器、Offer 沙盘、面试官 Skill 库、EvidenceGraph、FeedbackDistillation、分块流式输出、PDF 下载、人工反馈、无本地持久化 API，以及模型流式返回是否正常。

### 4.4 运行方式选择

| 场景 | 启动方式 | 页面 Base URL |
| --- | --- | --- |
| 本地只看 Mock Demo | `cd apps\web` 后执行 `python -m http.server 5173` | 留空 |
| 本地使用真实模型 | `python scripts\local_proxy.py --key-file 1.md` | `http://127.0.0.1:8787` |
| GitHub Pages 使用真实模型 | 部署 `serverless/cloudflare-worker.js` | 你的 Worker 根地址 |

## 5. 隐私与 Key

- 页面不需要登录。
- 页面不保存用户信息。
- API Key 只保存在当前页面内存。
- 刷新或关闭页面后，Key、简历、JD、报告都会丢失。
- 不要把真实 API Key 写入仓库。
- 不要在公共设备上输入真实简历和 API Key。

## 6. 主流模型 Key 获取方式

第一版使用“用户每次打开页面自行输入 Key”的方式。Key 只在当前浏览器页面内存中临时存在，刷新或关闭页面后会丢失。

### 6.1 通用获取流程

1. 注册并登录对应模型服务商的开放平台或控制台。
2. 完成必要的实名认证、账户绑定或付费额度开通。
3. 进入 `API Key`、`密钥管理`、`访问令牌` 或类似页面。
4. 创建新的 API Key。
5. 复制 Key，回到面试准备助手页面，在“临时 API Key”中粘贴。
6. 选择对应模型服务商和模型名称，点击生成报告。

### 6.2 常见服务商配置

| 服务商 | 常见入口 | 面试准备助手里如何选择 | 常见 Base URL | 常见模型名示例 |
| --- | --- | --- | --- | --- |
| DeepSeek | `platform.deepseek.com` → API Keys | `DeepSeek` | `https://api.deepseek.com/v1` | `deepseek-chat` |
| Kimi / Moonshot | `platform.moonshot.cn` → API Keys | `Kimi / Moonshot` | `https://api.moonshot.cn/v1` | `moonshot-v1-8k` |
| 阿里百炼 / 通义千问 | `bailian.console.aliyun.com` 或 DashScope 控制台 → API Key | `通义千问 / DashScope` | `https://dashscope.aliyuncs.com/compatible-mode/v1` | `qwen-plus` |
| OpenAI | `platform.openai.com` → API keys | `OpenAI` | `https://api.openai.com/v1` | `gpt-4.1-mini` |
| 豆包 / 火山方舟 | 火山引擎方舟控制台 → API Key | `OpenAI-Compatible 自定义接口` | `https://ark.cn-beijing.volces.com/api/v3` | 通常填写模型或 Endpoint ID |
| 智谱 GLM | `bigmodel.cn` 控制台 → API Keys | `OpenAI-Compatible 自定义接口` | `https://open.bigmodel.cn/api/paas/v4` | `glm-4-flash` 等 |
| 硅基流动 | `cloud.siliconflow.cn` → API Keys | `OpenAI-Compatible 自定义接口` | `https://api.siliconflow.cn/v1` | 平台展示的模型名 |
| 其他 OpenAI 兼容服务 | 对应服务商控制台 | `OpenAI-Compatible 代理 / 自定义接口` | 服务商提供的兼容地址或代理地址 | 服务商提供的模型名 |

### 6.3 填写建议

- 如果页面里已有对应服务商，例如 `DeepSeek`、`Kimi / Moonshot`、`通义千问 / DashScope`，优先直接选择对应项。
- 如果服务商没有出现在下拉框里，但兼容 OpenAI API，选择 `OpenAI-Compatible 代理 / 自定义接口`，并填写服务商提供的 Base URL。
- 如果页面提示 `Failed to fetch`，选择 `OpenAI-Compatible 代理 / 自定义接口`，并填写你自己部署的代理地址。
- 模型名称必须和服务商文档或控制台展示的一致，否则可能返回模型不存在。
- 不同服务商的免费额度、计费方式、上下文长度和跨域策略不同，以官方控制台说明为准。
- 如果浏览器直连报错，优先使用 `Mock Demo` 跑通流程；后续可自行增加代理或 Serverless Function。

### 6.4 安全提醒

- 不要把真实 Key 写入 `README.md`、`.env.example`、代码文件或 GitHub Issue。
- 不要把 Key 截图发到公开群或公开仓库。
- 如果怀疑 Key 泄露，立即到对应服务商控制台删除或重置。
- 建议为面试准备助手单独创建一个低额度 Key，避免误用主账户高权限 Key。
- 使用真实简历时，先确认模型服务商的数据使用和隐私政策。

## 7. 浏览器直连模型接口失败怎么办

部分模型服务商可能限制浏览器跨域请求。可以先使用 Mock Demo 跑通流程；后续如果需要真实稳定调用，可以自行加一个代理或 Serverless Function。

项目已经提供一个 Cloudflare Worker 示例：

```text
serverless/cloudflare-worker.js
```

### 7.1 部署 Cloudflare Worker 代理

1. 打开 Cloudflare Dashboard。
2. 进入 `Workers & Pages`。
3. 创建一个新的 Worker。
4. 将 `serverless/cloudflare-worker.js` 的内容复制到 Worker 编辑器。
5. 在 Worker 环境变量中新增：

```text
UPSTREAM_BASE_URL=https://ws-ppprfnb8819dlxt6.cn-beijing.maas.aliyuncs.com/compatible-mode/v1
```

6. 部署后得到 Worker 地址，例如：

```text
https://your-worker.your-name.workers.dev
```

7. 回到页面：
   - 模型服务商选择 `OpenAI-Compatible 代理 / 自定义接口`。
   - 模型名称填写 `qwen-plus`，或你的实际模型名。
   - 临时 API Key 填写你的模型 Key。
   - 代理 / 自定义 Base URL 填写 Worker 地址。

注意：Base URL 只填 Worker 根地址，不要加 `/chat/completions`。

## 8. 常见部署问题

### 8.1 `git remote add origin` 提示 remote 已存在

使用：

```powershell
git remote set-url origin https://github.com/<your-name>/<your-repo>.git
```

### 8.2 `git push` 要求登录

GitHub 现在通常不支持直接用账号密码推送。可以选择：

- 按 Git 弹出的浏览器登录流程授权。
- 或使用 GitHub Personal Access Token。

Token 创建路径通常是：

```text
GitHub → Settings → Developer settings → Personal access tokens
```

创建 Token 时至少需要能推送仓库代码的权限。

### 8.3 GitHub Pages 没有出现访问地址

检查：

1. 仓库是否已经推送到 `master`。
2. `Settings` → `Pages` 里 Source 是否选择了 `GitHub Actions`。
3. `Actions` 页面里的 workflow 是否成功。
4. 仓库是否包含 `.github/workflows/pages.yml`。

### 8.4 页面打开后样式丢失

检查仓库中是否存在：

```text
apps/web/index.html
apps/web/styles.css
apps/web/app.js
```

当前 workflow 会把 `apps/web` 作为静态站点目录上传，所以这三个文件必须在该目录下。

### 8.5 修改后页面没有更新

可能是浏览器缓存或 GitHub Pages 还没部署完成。

处理方式：

1. 到 GitHub `Actions` 确认最新部署已经成功。
2. 浏览器强制刷新。
3. 等待 1 到 3 分钟后再打开。

### 8.6 真实模型调用失败

可能原因：

- API Key 错误。
- 模型名填写错误。
- Base URL 不正确。
- 服务商限制浏览器跨域请求。
- 账户没有余额或额度。

建议先用 `Mock Demo` 跑通流程。如果 Mock 正常，但真实模型失败，说明页面本身基本可用，问题多半在模型服务商配置或浏览器跨域限制。

如果页面提示 `生成失败：Failed to fetch`，通常不是 Key 本身错误，而是浏览器在正式请求前的跨域预检被模型服务商拒绝。此时服务端接口可能可用，但 GitHub Pages 这类静态页面不能直接调用它。处理方式：

1. 先用 `Mock Demo` 验证页面流程。
2. 本地用 `python scripts\smoke_test.py --with-llm 1.md --model qwen-plus` 验证 Key 和接口是否可用。
3. 如果本地脚本可用但页面失败，说明是浏览器跨域限制，需要使用自己控制的代理或 Serverless Function。
4. 在页面的 `自定义 Base URL` 中填写代理后的 OpenAI-Compatible 地址。
