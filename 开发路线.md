# OfferAgent 开发路线：Offer 沙盘 + 面试官 Skill 库

## 1. 产品目标

OfferAgent 的最终产品形态明确为“Offer 沙盘 + 面试官 Skill 库”。它不是一个单纯的招聘辅助报告工具，也不是黑盒 AI 面试官，而是一个围绕候选人、岗位、公司、面试官和谈薪场景展开的可交互决策推演系统。

“Offer 沙盘”负责场景推演：基于“简历 + 岗位要求 + 项目角色 + 候选人回答证据 + 公司背景 + 面试反馈 + 薪资约束”，持续模拟面试准备、追问验证、候选人回答、风险校准、谈薪策略和录用决策。

“面试官 Skill 库”负责专业判断：沉淀 HR、业务面试官、技术面试官、岗位专家、公司画像、行业经验和谈薪专家的判断框架，让系统不只是生成一次性报告，而是积累可复用、可校准、可迭代的招聘认知资产。

第一版仍然收窄为“产品经理面试准备助手”。第一版优先帮助 HR、招聘经理和业务面试官在面试前快速理解候选人、识别风险点、生成基于项目证据的追问问题，并收集人工反馈。系统不替代人工录用判断，也不输出自动淘汰结论。

产品定位不是做一个“黑盒 AI 面试官”，而是做一个可解释的招聘辅助决策系统。它不仅要回答“候选人是否匹配”，还必须能讲清楚“为什么匹配、为什么不匹配、为什么认为有潜力”。每一个匹配结论、风险判断和潜力判断，都应该可以追溯到简历中的具体描述、岗位要求中的具体条目，或候选人在面试回答中的具体证据。

核心原则：

- 第一步先判断“是否匹配岗位要求”，不直接谈潜力。
- 第二步基于第一步发现的亮点、疑点和差距，再评估“未来潜力”。
- 两步走必须形成证据链：硬性匹配引用简历和 JD 证据，潜力判断引用项目追问和候选人回答证据。
- 报告必须区分事实、推断和待验证项，避免把模型猜测包装成确定结论。
- 面试官问题必须围绕候选人真实项目经历展开，避免泛泛而谈。
- 虚拟面试官需要有明确角色、关注维度和追问风格。
- 优先在特定岗位上做深，例如项目经理、产品经理、测试工程师、研发主管，再逐步扩展到通用招聘场景。
- 第一版先验证“面试问题是否有价值、证据链是否可信、用户是否愿意复用”，再验证自动面试和潜力评分。
- 长期产品路线围绕“Offer 沙盘 + 面试官 Skill 库”展开：沙盘提供动态交互，Skill 库沉淀专业判断，人工反馈闭环持续提高质量。

## 2. 核心用户流程

1. 用户输入候选人简历。
2. 用户输入岗位要求。
3. 系统抽取岗位关键词、能力要求、项目经验要求和隐性要求。
4. 系统解析简历中的技能、项目、职责、成果和候选人角色。
5. 系统为岗位要求和简历信息建立证据索引，记录每个判断来自哪里。
6. 系统完成第一步：简历硬性匹配，并输出匹配点、差距点、待验证点及对应证据。
7. 系统识别简历中的亮点、疑点、风险点和需要追问的项目。
8. 系统根据岗位类型、项目类型和候选人角色，生成虚拟面试官团队。
9. 虚拟面试官围绕项目进行结构化提问和多轮追问。
10. 系统把候选人回答沉淀为新的证据，更新项目真实性、贡献度和能力判断。
11. 系统完成第二步：潜力匹配，并明确每个潜力判断对应的回答证据和置信度。
12. 系统输出面试准备报告，包括岗位匹配、证据链、项目可信度、能力画像、风险点和必问追问。
13. 系统将本次评估保存为一次 Evaluation Run，为后续 Offer 沙盘推演提供上下文。
14. 后续版本中，系统可以继续进入 Offer Simulation Run，围绕面试表现、面试官反馈、候选人动机、竞争 Offer 和薪资预算推演录用与谈薪策略。

## 3. MVP 版本范围

MVP 阶段先做成一个可用的文本评估工具，不追求复杂前端。MVP 的核心不是证明系统“能生成报告”，而是验证招聘团队是否认可“产品经理岗位解析 + 证据链 + 面试前追问清单 + 人工反馈”的工作流。

MVP 需要直面三个关键不确定性：

- 系统的硬性匹配结论是否接近资深招聘方或资深面试官的判断。
- 系统生成的产品经理追问问题是否真的能帮助业务面试官识别候选人的真实贡献。
- 证据链是否能让 HR、招聘经理和业务面试官相信系统不是黑盒判断。
- 真实用户是否愿意使用、复用并推荐这个工具。

建议 MVP 第一版先聚焦产品经理岗位做测试和打磨产品，不建议一开始覆盖多个岗位。第一版的重点不是证明系统可以适配所有岗位，也不是直接做完整 AI 面试，而是围绕产品经理的真实招聘场景，把“硬性匹配 + 证据链 + 项目追问 + 人工反馈”的闭环跑通，再决定是否扩展到项目经理、测试工程师或研发主管等岗位。

第一版优先服务“业务面试前准备”这个节点。系统输出的核心交付物应该是：

- 候选人与产品经理岗位的硬性匹配摘要。
- 产品项目经历中的亮点、风险点和待验证点。
- 每个关键判断对应的 1 到 3 条核心证据。
- 面试官可直接采用或改写的 3 到 5 个必问问题。
- 人工反馈入口，用于记录同意、不同意、问题是否被采用和面试后是否验证风险。

第一版暂不做：

- 多岗位模板。
- 候选人实时在线面试。
- 多轮自动追问。
- ATS 集成。
- 候选人横向对比。
- 复杂权限系统。
- 精美 PDF 报告。
- 完整 Offer 沙盘。
- 复杂 Skill 市场。

但第一版的数据结构和系统边界需要为最终形态预留扩展点，包括 Evaluation Run、Offer Simulation Run、Role Agent Layer、Skill Registry、Evidence Graph 和 Feedback Distillation。

### 3.0 当前 Web MVP 完成状态

截至当前版本，项目已经完成一个可演示、可本地运行、可 GitHub Pages 部署的纯静态 Web MVP。当前实现不是完整招聘系统，也不是持久化评估平台，而是围绕“简历 + JD + 面试上下文”的面试前准备闭环，完成了从输入、Mock / 真实模型生成、候选人与面试官报告拆分、Offer 推演、人工反馈到 PDF 导出的前端产品化。

已完成的主链路：

- 输入侧：支持简历、岗位 JD、公司 / 面试上下文、候选人阶段、目标职级、Offer / 谈薪约束和面试官视角选择。
- 生成侧：支持本地 Mock Demo，也支持用户临时填写 API Key 与 Base URL 调用 OpenAI-Compatible 接口。
- 报告侧：已经拆分候选人报告、面试官报告和 Offer 沙盘推演报告，并提供报告预览。
- 证据侧：在前端规则层完成岗位要求与简历证据的初步映射，输出一级 / 二级 / 三级证据可信度、缺证项和验证问题。
- 闸口侧：支持“匹配进入 / 条件性进入（转岗适配） / 不匹配不推进”三类项目匹配闸口，并输出能力迁移话术和下一轮验证建议。
- 面试官侧：已内置 HR、业务负责人、项目推进、谈薪顾问、决策层压力官等视角，并固定强化决策层压力官的压力测试问题。
- Offer 侧：已生成 Offer 推演报告，覆盖证据可信度、谈判杠杆、职级 / 薪资 / 到岗约束、风险项和后续推进建议。
- 反馈侧：支持在当前报告中追加人工反馈，包括是否同意判断、问题是否有效和补充备注。
- 导出侧：支持候选人 PDF、面试官 PDF、Offer 推演 PDF 三类浏览器端导出。
- 国际化侧：支持中文 / English 双语切换，页面 UI、Mock 报告、真实模型提示、报告预览和 PDF 输出会跟随语言选择切换。
- 工程侧：已提供 GitHub Pages 部署说明、Cloudflare Worker 代理示例、本地代理脚本、smoke test 脚本、EvaluationRun Schema 和 SkillDefinition Schema。

当前仍需注意的边界：

- 证据可信度分级目前主要由前端规则、关键词和模型输出共同支撑，还不是稳定的结构化评分引擎。
- EvaluationRun 和 SkillDefinition 已有 Schema 草案，但 Web MVP 尚未接入持久化数据库或长期版本管理。
- 人工反馈只追加到当前报告，尚未沉淀为可检索、可统计、可蒸馏的反馈数据集。
- 面试官视角库目前是内置固定视角，不是基于岗位、公司和候选人动态生成的 Skill Registry。
- Offer 沙盘目前是报告式推演，不是可交互、多轮更新、可保存状态的 Offer Simulation Run。
- 当前 Web 实现集中在 `apps/web/app.js`，后续需要按输入、生成、报告、PDF、i18n、证据和沙盘逻辑拆分模块。

因此，当前项目已经从“路线设计”进入“可演示 MVP”阶段。后续开发重点不应继续堆报告模块，而应转向真实样本评测、结构化数据沉淀、人工反馈闭环和代码模块化。

### 3.1 输入模块

- 简历文本输入。
- 岗位要求文本输入。
- 可选输入：目标职级、面试方向、公司类型、是否偏技术或管理。

### 3.2 简历解析模块

需要抽取以下信息：

- 基本背景：年限、行业、学历、公司经历。
- 技能栈：语言、框架、数据库、中间件、云服务、工具链。
- 项目经历：项目名称、业务背景、技术方案、职责、成果。
- 候选人角色：主导、负责、参与、协作、支持。
- 可量化成果：性能提升、成本下降、稳定性提升、营收影响、团队规模。

### 3.3 岗位解析模块

需要抽取以下信息：

- 必备技能。
- 加分技能。
- 核心职责。
- 项目经验要求。
- 管理或协作要求。
- 隐性要求，例如抗压、owner 意识、跨部门推动、架构能力、业务理解。

### 3.4 第一步：硬性匹配模块

输出内容：

- 总体匹配结论：高度匹配、基本匹配、部分匹配、差距较大。
- 技能匹配表。
- 项目经验匹配表。
- 职责匹配表。
- 明显亮点。
- 明显缺口。
- 简历疑点。
- 每条匹配结论对应的证据来源。
- 每条不确定结论对应的验证问题。

关键规则：

- 只基于简历和岗位要求做判断。
- 不因为候选人表达好就提前判断潜力。
- 对“参与、负责、主导”等词做贡献度拆解。
- 对没有证据支撑的成果标记为待验证。
- 没有证据的结论不得进入“已匹配”，只能进入“待验证”。
- 第一阶段输出必须能回答：这条判断来自简历哪句话、岗位哪条要求。

### 3.5 产品经理岗位能力矩阵

第一版需要单独建立产品经理岗位能力矩阵，所有解析、追问和报告都围绕该矩阵展开。

核心能力维度：

- 需求洞察：是否能识别真实用户问题，而不是只接收业务方需求。
- 业务理解：是否知道项目服务哪个业务目标，能否说清楚业务约束和收益逻辑。
- 指标意识：是否能定义核心指标、解释指标口径，并区分相关性和因果关系。
- 优先级判断：是否能在资源有限时做取舍，并说明取舍依据。
- 跨团队推动：是否能推动研发、设计、运营、销售或客户方协同。
- 用户价值判断：是否能把功能设计与用户价值、业务价值连接起来。
- 项目复盘：是否能复盘成功或失败原因，而不是只陈述结果。
- 沟通表达：是否能清晰还原背景、问题、决策、行动和结果。

每个能力维度都需要定义：

- 高质量证据是什么。
- 弱证据是什么。
- 常见包装话术是什么。
- 应该如何追问。
- 对应的风险回答特征。

### 3.6 第二步：潜力相关行为证据模块

基于第一步结果生成面试官团队。

第二步不是重新评价候选人，也不是直接输出高精度潜力评分，而是沿着第一步发现的亮点、差距和疑点继续验证。第一版应把“潜力评估”改为“潜力相关行为证据与待验证假设”。这些判断只能来自候选人在项目追问中的行为证据，例如如何拆解问题、如何做取舍、如何复盘失败、如何推动他人协作。

第一版报告中应避免把潜力写成确定结论，建议使用：

- 已有较强证据。
- 有一定证据但需要继续验证。
- 暂无足够证据。
- 存在风险信号。

默认虚拟面试官：

- 项目经理：关注交付、进度、资源协调、风险控制、跨部门沟通。
- 研发主管：关注技术深度、架构判断、工程质量、复杂问题解决。
- HRBP / 潜力星探：关注学习敏锐度、自驱力、复盘能力、文化契合度。

可扩展面试官：

- 架构师：关注系统设计、扩展性、稳定性、技术选型。
- 业务负责人：关注业务理解、指标意识、用户价值。
- 测试负责人：关注质量意识、异常场景、上线风险。
- 数据负责人：关注数据口径、指标归因、实验设计。

### 3.7 面试追问模块

每个面试官生成：

- 3 到 5 个核心问题。
- 每个问题配 2 到 3 个追问。
- 每个问题标注评估维度。
- 每个问题说明优秀回答特征和风险回答特征。
- 第一版默认从候选问题中筛选 3 到 5 个必问问题，而不是让面试官照搬所有问题。

问题类型：

- 项目真实性追问。
- 贡献度追问。
- 决策逻辑追问。
- 冲突处理追问。
- 失败复盘追问。
- 技术权衡追问。
- 结果归因追问。

### 3.8 评估报告模块

报告结构：

- 候选人与岗位硬性匹配结论。
- 两步走证据链摘要。
- 项目经历可信度判断。
- 各虚拟面试官评价。
- 四类潜力维度评分：
  - 思维模型。
  - 变革敏锐度。
  - 人际影响力。
  - 结果驱动力。
- 风险点。
- 建议继续追问的问题。
- 与资深面试官人工判断的差异记录。
- 录用建议：强烈推荐、推荐、谨慎推进、不推荐。

第一版报告需要控制信息密度，默认只展示结论和最关键的 1 到 3 条证据，完整证据链放在详情层。报告首页应能在 1 分钟内让用户看到最大亮点、最大风险和必问问题。

### 3.9 MVP 成功指标

第一版必须用真实使用指标判断是否继续推进，而不是只看功能是否完成。

核心指标：

- 硬性匹配与人工判断一致率：10 份样本中至少 7 份基本一致。
- 系统追问被采用率：超过 50% 的问题被实际采用或改写采用。
- 面试官准备时间节省比例：目标节省 30% 以上。
- 报告可读性评分：HR 或业务面试官能在 3 分钟内抓住候选人重点。
- 证据链可信度评分：用户认为关键结论有依据，而不是主观点评。
- 高频误判类型：记录并能归因到证据遗漏、权重不同、岗位理解不同或系统误判。
- 同一用户重复使用率：至少 30% 的试用用户在两周内重复使用。
- 用户推荐意愿：至少 3 位真实招聘参与者愿意推荐给其他 HR 或面试官。
- 付费意愿访谈结果：至少 3 位用户表达明确付费、试点或企业内部推荐意愿。

## 4. 分阶段开发路线

### 阶段一：规则化原型

目标：快速验证“产品经理岗位解析 + 证据链解释 + 面试前追问清单 + 人工反馈”是否可被招聘团队接受。

功能：

- 支持粘贴简历和岗位要求。
- 用固定 prompt 完成简历解析和岗位解析。
- 输出第一步硬性匹配结果。
- 输出默认三位虚拟面试官的问题清单。
- 生成一版静态评估报告。
- 每条匹配结论必须绑定简历或 JD 原文证据。
- 每个潜力相关行为假设必须绑定候选人项目经历或模拟回答证据。
- 第一版固定支持产品经理岗位模板，后续再扩展项目经理、测试工程师或研发主管等岗位。
- 输出 3 到 5 个面试前必问问题，并说明为什么问、验证什么、依据来自哪里。
- 支持人工标注问题是否被采用、结论是否同意、证据是否充分。

交付物：

- Prompt 模板。
- 基础命令行或简单 Web 页面。
- 一份完整示例报告。
- 证据链展示格式。
- 产品经理岗位能力矩阵。
- 产品经理岗位追问模板。
- 与资深面试官人工判断的对照记录表。
- 人工反馈表。

验收标准：

- 能清晰区分“岗位匹配”和“潜力相关行为证据”。
- 能从项目经历中生成具体追问，而不是生成通用八股问题。
- 能识别简历中“主导、负责、参与”背后的待验证点。
- 每个关键结论都能追溯到证据。
- 招聘方能看懂系统为什么得出该结论。
- 与资深面试官判断不一致时，系统能暴露差异原因，而不是只给不同分数。
- 系统生成的问题被业务面试官实际采用或改写采用的比例超过 50%。
- 报告首页能在 1 到 3 分钟内帮助 HR 或业务面试官抓住候选人重点。

### 阶段二：结构化数据与评分体系

目标：让评估结果更稳定、可比较、可解释。

功能：

- 将简历解析为结构化 JSON。
- 将岗位要求解析为结构化 JSON。
- 建立技能、职责、项目、潜力四类评分模型。
- 支持评分理由引用原文证据。
- 支持按岗位类型调整权重。
- 建立 Evidence Schema，统一管理简历证据、JD 证据、回答证据和人工反馈证据。
- 建立垂直岗位评分权重，例如项目经理更重交付和协同，产品经理更重需求判断和业务指标，测试工程师更重质量风险和异常场景。
- 支持人工面试官对系统结论进行标注：同意、部分同意、不同意、需要补充证据。

交付物：

- Resume Schema。
- Job Requirement Schema。
- Evidence Schema。
- Evaluation Schema。
- 评分规则说明。
- 岗位垂直化权重配置。
- 人工评审反馈表。

验收标准：

- 同一份简历和岗位多次评估结果稳定。
- 每个评分都有证据来源。
- 缺少证据时系统明确标注“不确定”，而不是编造判断。
- 不同岗位使用不同权重和追问策略。
- 能统计系统判断与资深面试官判断的一致率和分歧点。

### 阶段三：岗位垂直化追问模型

目标：在特定岗位上做深，形成区别于通用 AI 面试官的场景适配能力。

优先岗位：

- 项目经理：重点验证项目交付、风险管理、资源协调、干系人沟通、复盘能力。
- 产品经理：重点验证需求洞察、业务目标、优先级判断、数据意识、跨团队推动。
- 测试工程师：重点验证质量策略、异常场景、自动化测试、上线风险、缺陷归因。
- 研发主管：重点验证技术决策、架构权衡、团队管理、工程质量、复杂问题解决。

功能：

- 每类岗位维护独立的能力模型。
- 每类岗位维护独立的虚拟面试官组合。
- 每类岗位维护独立的追问策略和优秀回答特征。
- 根据候选人在项目中的角色选择追问深度。
- 对岗位内常见包装话术建立验证问题库。

交付物：

- 项目经理岗位模板。
- 产品经理岗位模板。
- 测试工程师岗位模板。
- 研发主管岗位模板。
- 垂直岗位问题库。
- 岗位能力矩阵。

验收标准：

- 同一候选人在不同岗位下会得到不同追问，而不是通用问题复用。
- 业务面试官认为问题能覆盖该岗位的关键风险。
- 对项目角色的追问能区分负责人、核心参与者和普通参与者。

### 阶段四：交互式虚拟面试

目标：从静态问题清单升级为可对话面试。

功能：

- 候选人逐题回答。
- 虚拟面试官根据回答自动追问。
- 不同面试官保留自己的关注维度和风格。
- 系统根据回答实时更新风险点和潜力判断。
- 支持中途切换面试官。

交付物：

- 面试会话状态管理。
- 面试官角色配置。
- 追问策略。
- 回答评分器。

验收标准：

- 追问能引用候选人刚才的回答。
- 面试官不会偏离候选人的项目经历。
- 能把“回答流畅”与“真实贡献”区分开。
- 回答产生的新证据能自动进入证据链。
- 第二步潜力评估能解释它引用了哪些回答。

### 阶段五：报告与招聘决策辅助

目标：服务真实招聘决策。

功能：

- 生成标准化面试报告。
- 生成面试官纪要。
- 输出候选人风险雷达图或能力矩阵。
- 输出下一轮面试建议。
- 支持同岗位多个候选人横向对比。
- 输出“系统判断 vs 人工判断”差异分析。
- 输出每个结论的证据链和置信度。
- 支持招聘方对报告结论进行反馈，形成后续优化数据。

交付物：

- PDF / Markdown 报告。
- 候选人对比视图。
- 面试结论模板。
- 证据链详情页。
- 人工反馈与差异分析页。

验收标准：

- 报告能直接给招聘团队使用。
- 能明确说明推荐或不推荐的依据。
- 能指出下一轮面试需要验证的关键问题。
- 招聘团队能通过报告复盘每个关键判断。
- 报告能支持“辅助决策”，而不是只做“辅助筛选”。

### 阶段六：产品化与团队协作

目标：让系统可在招聘流程中持续使用。

功能：

- 用户登录和权限管理。
- 岗位库。
- 候选人库。
- 面试记录归档。
- 评估标准版本管理。
- 团队协作评论。
- 与 ATS 或招聘系统集成。

交付物：

- Web 应用。
- 数据库设计。
- API 接口。
- 权限模型。

验收标准：

- 支持真实招聘团队多人使用。
- 历史面试记录可追溯。
- 岗位标准和评估结果可复盘。

## 5. 推荐技术架构

最终架构应围绕“Offer 沙盘 + 面试官 Skill 库”设计。第一版可以只实现其中最小可用部分，但数据结构、模块边界和日志记录要为后续沙盘化与 Skill 化预留空间。

### 5.1 前端

- React / Next.js。
- 简历与岗位输入页面。
- 面试官配置页面。
- 交互式面试页面。
- 评估报告页面。
- 后续扩展 Offer 沙盘工作台：支持面试前准备、风险追问、候选人回答模拟、谈薪推演和决策复盘。
- 后续扩展 Skill 管理页面：支持岗位 Skill、公司 Skill、面试官 Skill、谈薪 Skill 的版本、边界和适用场景管理。

### 5.2 后端

- Python FastAPI 或 Node.js NestJS。
- LLM 调用服务。
- 文档解析服务。
- 评分与报告生成服务。
- 面试会话状态服务。
- Skill Registry 服务。
- Offer Simulation 服务。
- Feedback Distillation 服务。

### 5.3 数据库

- PostgreSQL：存储岗位、候选人、评估报告、面试记录。
- Redis：存储临时会话和异步任务状态。
- 对象存储：保存原始简历文件和报告文件。
- 图谱或关系扩展：保存候选人、岗位、项目、证据、问题、反馈、面试官角色和 Skill 之间的关系。

### 5.4 LLM 能力层

- 简历结构化抽取。
- 岗位要求结构化抽取。
- 匹配度分析。
- 虚拟面试官生成。
- 追问生成。
- 回答评分。
- 报告生成。
- 面试官 Skill 调用。
- Offer 场景推演。
- 谈薪策略模拟。
- 人工反馈蒸馏。

### 5.5 GitHub 托管策略

最小 Web MVP 第一版按开源个人工具设计，可以先放在 GitHub 上托管。第一版不按商业 SaaS 设计，不需要用户系统、数据库、服务端长期存储或企业级权限。

推荐方式：

- 前端使用 React / Next.js 静态导出，托管到 GitHub Pages。
- 第一版可以直接开源到 GitHub，定位为可自用、可 Fork、可本地运行的开源 MVP。
- 使用 GitHub Actions 自动构建和部署前端页面。
- 简历、JD、Key 和报告结果只在当前浏览器页面运行时内存中临时存在，不上传、不存库、不写数据库。
- 大模型 API Key 采用用户每次打开页面自行设置模式，系统不内置、不托管、不默认提供模型 Key。
- 用户刷新页面或关闭页面后，Key 自动丢失，再次使用需要重新设置。
- 前端可以提供临时 Key 输入框和模型服务商选择，但默认不写入 LocalStorage、SessionStorage、Cookie 或远程服务。
- 如果用户担心浏览器直连模型服务暴露 Key，可选择本地运行或后续自行配置代理服务；第一版默认以开源个人工具和 GitHub Pages 演示为主。
- 如果只是早期演示，可以先使用 Mock 数据或手动粘贴模型输出，验证页面流程和报告结构。

第一版推荐部署形态：

```text
GitHub Repo
├── apps/web               # 最小 Web MVP 前端
├── docs                   # 产品文档与示例报告
├── prompts                # Prompt 与 Skill 草案
├── examples               # 脱敏简历、JD、报告样例
└── schemas                # CandidateProfile、JobProfile、Evidence、EvaluationRun 等 Schema
```

GitHub Pages 适合托管：

- 产品介绍页。
- 简历和 JD 文本输入界面。
- Mock 分析结果页面。
- 示例报告展示。
- 不含敏感数据的 Demo。
- 用户本地临时配置 Key 的纯前端开源工具。
- 报告生成后下载为 Markdown / JSON / HTML 文件。

GitHub Pages 不适合直接承担：

- LLM API Key 长期保存。
- 大模型 Key 托管与密钥管理。
- 真实简历长期存储。
- 用户登录和权限系统。
- 企业级数据安全和审计。
- 私密招聘数据处理。

因此，GitHub 托管适合作为最小 Web MVP 的展示和验证入口。第一版不保存用户信息，不保存 Key，不保存简历，不保存报告；用户每次打开页面自行设置 Key，生成报告后下载到本地，刷新页面后需要重新设置。

第一版前端临时配置建议：

```text
LLM_PROVIDER=openai | deepseek | anthropic | qwen | kimi | custom
LLM_MODEL=
TEMP_LLM_API_KEY=用户每次打开页面手动输入，仅保存在当前页面内存
CUSTOM_LLM_BASE_URL=可选，仅当前页面内存
```

Key 配置原则：

- 第一版不要求用户登录。
- 第一版不保存任何用户信息。
- 第一版不保存简历、JD、报告和反馈数据。
- Key 只保存在页面运行时内存中，刷新、关闭页面后立即丢失。
- 不使用 LocalStorage、SessionStorage、Cookie、IndexedDB 保存 Key。
- 不提交真实 Key，不提供默认 Key，不在日志、错误信息和报告中输出完整 Key。
- 如果用户没有配置 Key，系统应进入 Mock Demo 模式，允许查看示例流程但不调用真实模型。
- 报告生成后提供下载按钮，优先支持 Markdown，其次支持 JSON / HTML。
- 需要复盘时，用户自己保存下载文件；系统不提供云端历史记录。

开源仓库建议：

- 提供清晰 README，说明这是开源个人工具，不是商业招聘系统。
- 提供 `LICENSE`，建议使用 MIT 或 Apache-2.0。
- 提供 `.env.example` 仅用于可选本地开发，不包含真实 Key。
- 提供隐私说明：所有输入默认仅在浏览器当前会话中处理，刷新后清空。
- 提供安全提示：不要在公共设备输入真实简历和 API Key。
- 提供示例数据：只使用脱敏简历、脱敏 JD 和 Mock 报告。

## 6. Prompt 模板规划

### 6.1 简历解析 Prompt

目标：把非结构化简历转为结构化候选人画像。

输出字段：

- 候选人基本信息。
- 技能标签。
- 项目列表。
- 每个项目中的角色判断。
- 每个项目的可验证成果。
- 简历疑点。

### 6.2 岗位解析 Prompt

目标：把岗位 JD 转为结构化要求。

输出字段：

- 必备条件。
- 加分条件。
- 核心职责。
- 技术栈。
- 项目经验。
- 管理协作要求。
- 隐性能力要求。

### 6.3 硬性匹配 Prompt

目标：先评估候选人是否满足岗位要求。

约束：

- 不评估潜力。
- 不根据想象补全经历。
- 每条判断必须引用简历或岗位原文依据。

### 6.4 虚拟面试官生成 Prompt

目标：根据候选人项目角色和岗位要求生成合适的面试官。

输出字段：

- 面试官姓名。
- 面试官角色。
- 核心关注维度。
- 提问风格。
- 需要验证的问题。

### 6.5 潜力相关行为证据 Prompt

目标：基于候选人项目经历或回答，提取与成长性、复杂问题处理和跨团队推动相关的行为证据，形成待验证假设，而不是直接输出确定性潜力评分。

评估维度：

- 思维模型。
- 变革敏锐度。
- 人际影响力。
- 结果驱动力。
- 学习敏锐度。
- Owner 意识。

约束：

- 不输出自动录用或淘汰结论。
- 不把表达流畅、名校、大厂背景直接等同于潜力。
- 每个判断必须引用项目经历或候选人回答证据。
- 缺少证据时只能标注为“待验证”，不得包装成确定结论。

### 6.6 面试官 Skill Prompt

目标：把 HR、业务面试官、技术面试官、岗位专家、公司画像和谈薪专家的判断框架沉淀为可复用 Skill，而不是一次性角色扮演。

每个 Skill 至少包含：

- 适用场景。
- 核心判断框架。
- 高质量证据标准。
- 常见误判。
- 推荐追问。
- 反模式。
- 输出边界。
- 置信度规则。
- 人工反馈字段。

优先建设的 Skill：

- HR Skill：关注流程效率、候选人沟通、风险记录、合规边界和招聘推进。
- 业务面试官 Skill：关注岗位胜任力、项目真实性、业务判断、协作方式和落地能力。
- 产品经理岗位 Skill：关注需求洞察、业务理解、优先级判断、数据意识、跨团队推动和复盘能力。
- 技术面试官 Skill：关注技术深度、工程质量、架构判断、问题拆解和协作风险。
- 谈薪 Skill：关注候选人动机、市场薪酬、竞争 Offer、让步边界和沟通话术。
- 公司 Skill：沉淀某类公司或具体公司的业务阶段、组织风格、面试偏好和用人标准。
- 行业 Skill：沉淀不同行业对岗位能力的特殊要求，例如 SaaS、AI 应用、跨境电商、金融科技等。

### 6.7 Offer 沙盘 Prompt

目标：围绕一个候选人和一个岗位，模拟面试、追问、风险验证、谈薪和录用决策的关键路径。

输入上下文：

- 候选人结构化画像。
- 岗位结构化画像。
- 公司背景和业务阶段。
- 面试问题与候选人回答。
- 面试官反馈。
- 薪资范围、候选人期望和竞争 Offer 信息。
- 已有证据链和不确定项。

输出内容：

- Offer 推进倾向。
- 关键风险与待验证问题。
- 面试官分歧点。
- 候选人动机判断。
- 谈薪策略建议。
- 候选人可能接受或拒绝 Offer 的原因。
- 下一步行动建议。

约束：

- 不承诺预测真实 Offer 结果。
- 不输出自动录用或自动淘汰结论。
- 不把候选人画像包装成确定事实。
- 每个关键判断必须绑定证据、假设和置信度。

## 8. 风险与注意事项

- 不要把虚拟面试官做成只会问通用问题的聊天机器人。
- 不要在第一步就因为候选人“看起来聪明”而高估潜力。
- 不要把简历中的“参与”自动等同于“主导”。
- 不要编造候选人没有提供的经历。
- 对模型不确定的判断必须显式标注。
- 评估报告应作为招聘辅助，不应替代最终人工判断。
- 第一版不输出“拒绝录用”或“自动淘汰”结论，只输出风险点、证据和待验证问题。
- 默认不使用年龄、性别、婚育、照片、籍贯等敏感信息作为判断依据。
- 联系方式等个人信息应默认脱敏，测试和演示优先使用脱敏简历。
- 对外优先定位为“产品经理面试准备助手”，避免让用户误解为系统替 HR 做最终招聘决策。
- 对外长期定位可以逐步升级为“Offer 沙盘 + 面试官 Skill 库”，但必须明确它是决策辅助和场景推演系统，不是自动录用系统。
- 不模拟或冒充真实具体面试官本人，除非信息来自用户授权材料且明确标注为推测。
- 不承诺精准预测真实 Offer 结果，只提供准备、推演和风险校准。

## 9. 需求深挖：难点与风险点

这个系统的难点不在于“生成几个面试问题”，而在于能否把简历、岗位要求、项目角色、候选人回答和最终判断之间的证据链建立起来。真正有价值的评估系统必须做到：先判断岗位匹配，再围绕候选人实际项目追问，最后谨慎推断潜力。

### 9.1 业务本质难点

#### 9.1.1 岗位要求经常是不完整或失真的

很多 JD 并不能真实表达岗位需求。常见情况包括：

- JD 写了很多技术栈，但真正关键的是业务复杂度和交付经验。
- JD 写的是高级工程师，实际希望招的是能带项目的小组负责人。
- JD 强调“沟通能力”，但没有说明是跨部门推动、客户沟通，还是团队协作。
- JD 把必备项和加分项混在一起，导致系统难以判断权重。
- 不同公司对同一个职级的能力要求差异很大。

产品难点：

- 需要把 JD 拆成“显性要求”和“隐性要求”。
- 需要允许用户确认或调整岗位权重。
- 不能只做关键词匹配，否则会高估简历包装能力强的人。

风险：

- 如果岗位解析错了，后续匹配和面试问题都会偏。
- 如果系统不能解释“为什么这样理解岗位”，招聘方很难信任结果。

缓解方式：

- 在岗位解析后增加确认页，让用户确认必备项、加分项和权重。
- 对不清晰的岗位要求标注“需要人工确认”。
- 支持岗位模板，例如后端工程师、项目经理、研发主管、数据分析师等。

#### 9.1.2 简历信息天然带有包装和噪音

候选人简历常见问题：

- 用“负责”包装“参与”。
- 用团队成果包装个人贡献。
- 只写技术名词，不写决策过程。
- 只写项目结果，不写上下文和约束。
- 项目经历缺少失败、冲突、取舍和复盘。
- 成果数据没有口径，例如“性能提升 50%”不知道提升的是接口耗时、吞吐量还是页面加载速度。

产品难点：

- 系统不能把简历描述直接当事实。
- 需要判断“证据强度”，而不是只判断“是否出现关键词”。
- 需要区分“候选人声称的角色”和“从描述中推断的真实角色”。

风险：

- 过度相信简历会导致误判。
- 过度怀疑简历会导致误伤优秀候选人。
- 如果问题太像审问，候选人体验会变差。

缓解方式：

- 每个项目增加证据等级：高、中、低。
- 对“主导、负责、参与、协作”建立贡献度验证规则。
- 面试问题采用“请还原当时场景”的方式，而不是直接质疑。

#### 9.1.3 “潜力”很难被直接测量

潜力不是简历上的事实，而是基于行为证据的推断。系统需要避免把以下因素误判为潜力：

- 表达能力强。
- 熟悉面试套路。
- 简历写得漂亮。
- 名校或大厂背景。
- 对流行概念掌握较多。

真正更接近潜力的证据包括：

- 面对未知问题时是否能结构化拆解。
- 遇到变化时是否主动调整策略。
- 是否能从失败中提炼可复用经验。
- 是否能在资源不足时推动结果。
- 是否能把业务目标、技术方案和组织协作连起来。

产品难点：

- 潜力评估必须建立在具体项目行为上。
- 不能只通过一次回答给出武断结论。
- 需要区分“当前能力不足”和“成长速度很快”。

风险：

- 潜力评估容易主观化。
- 模型可能受到候选人表达风格影响。
- 如果报告写得太绝对，会影响招聘公平性。

缓解方式：

- 潜力评分必须引用候选人的回答证据。
- 使用“倾向判断”而不是绝对判断。
- 给出置信度，例如高置信、中置信、低置信。

### 9.2 核心产品难点

#### 9.2.1 两步评估顺序必须被产品强制保证

这个系统的关键设计是：

1. 第一步先做硬性匹配。
2. 第二步再做潜力匹配。

难点在于模型很容易提前进入“综合评价”，把匹配和潜力混在一起。例如：

- 简历硬条件不够，但候选人看起来有潜力，系统提前给高推荐。
- 简历硬条件很好，但项目贡献度不清楚，系统直接推荐。
- 岗位要求是高级研发主管，候选人只做过执行型开发，系统因为技术栈匹配而忽略管理差距。

产品风险：

- 招聘方会误以为这是一个“智能打分器”，忽略证据边界。
- 候选人可能因为潜力描述被过度拔高或低估。

设计要求：

- 第一阶段报告中禁止出现潜力评分。
- 第二阶段必须引用第一阶段的亮点、差距和疑点。
- 最终报告要分开呈现“当前匹配度”和“未来潜力”。

#### 9.2.2 虚拟面试官不能只是换语气

“项目经理、研发主管、HRBP”不能只是名字不同，问题必须真的不同。

项目经理应该追问：

- 目标怎么定的。
- 进度如何拆解。
- 风险如何提前发现。
- 资源冲突怎么处理。
- 交付失败时怎么补救。

研发主管应该追问：

- 技术方案为什么这样选。
- 替代方案是什么。
- 架构边界在哪里。
- 性能、稳定性、可维护性如何权衡。
- 代码质量和工程规范如何保证。

HRBP / 潜力星探应该追问：

- 为什么选择承担这个角色。
- 面对压力如何调整。
- 从失败中学到了什么。
- 如何接受反馈。
- 长期职业驱动力是什么。

产品难点：

- 面试官角色需要配置化，而不是硬编码。
- 面试官问题要基于候选人的项目和岗位要求生成。
- 不同面试官之间需要互补，避免重复问同一类问题。

风险：

- 如果问题重复，用户会觉得系统只是套模板。
- 如果问题太发散，会偏离岗位评估目标。
- 如果问题太尖锐，可能造成不友好的候选人体验。

缓解方式：

- 为每个面试官定义关注维度、问题边界和禁问范围。
- 生成问题后做去重和覆盖度检查。
- 每个问题都绑定一个评估目标。

#### 9.2.3 项目角色识别是评估质量的核心

用户明确提到“根据面试者在项目中承担的角色”，这说明系统必须围绕候选人的真实项目角色设计面试。

项目角色可能包括：

- 项目负责人。
- 技术负责人。
- 核心开发。
- 模块开发。
- 协作支持。
- 需求分析。
- 测试或交付支持。
- 跨部门推动者。

难点：

- 简历中的角色描述往往模糊。
- 同一个项目中候选人可能有多个角色。
- 不同角色对应不同的评估重点。

例子：

- 如果候选人是项目负责人，应重点追问目标拆解、风险控制、资源协调和结果复盘。
- 如果候选人是技术负责人，应重点追问技术选型、架构权衡、性能稳定性和工程质量。
- 如果候选人是模块开发，应重点追问边界理解、实现细节、异常处理和协作方式。
- 如果候选人只是参与者，不应直接用负责人标准评估管理能力。

风险：

- 角色识别错了，问题会问偏。
- 用高级角色标准评估参与者，会不公平。
- 用低阶角色标准评估负责人，会低估候选人。

缓解方式：

- 对每个项目输出“角色声明”和“角色推断”。
- 对角色置信度低的项目优先追问确认。
- 面试官问题按角色分层生成。

### 9.3 模型与算法难点

#### 9.3.1 证据链构建

系统每个判断都应该能回答：

- 依据来自简历哪一句。
- 依据来自岗位要求哪一条。
- 依据来自候选人哪次回答。
- 这个判断的置信度是多少。
- 这是事实、推断，还是待验证假设。
- 如果与人工面试官判断不同，差异来自证据不足、权重不同，还是解释不同。

难点：

- 简历和 JD 是非结构化文本。
- 同一能力可能分散在多个项目中。
- 候选人回答可能补充、推翻或弱化简历信息。
- 潜力判断往往不是单条证据可以支撑，而是多个行为证据的组合推断。

风险：

- 没有证据链，报告会像主观点评。
- 模型可能产生听起来合理但没有依据的结论。
- 证据链如果展示得太复杂，招聘方会看不懂。
- 如果只展示结论不展示证据，产品会退化成黑盒打分器。

缓解方式：

- 使用结构化中间结果。
- 每个评分字段都要求 evidence。
- 报告中区分“事实、推断、待验证”。
- 为每个关键结论展示 1 到 3 条最关键证据，而不是把所有证据堆给用户。
- 对没有证据的内容只允许进入“待验证问题”，不允许进入最终判断。

#### 9.3.2 岗位垂直化能力模型

通用 AI 面试官容易问出“看似合理但没有岗位针对性”的问题。真正形成壁垒的方向，是在特定岗位上做深，让系统理解该岗位的真实评价逻辑。

优先建议做深的岗位：

- 项目经理：交付目标、风险控制、资源协调、干系人管理、项目复盘。
- 产品经理：需求判断、用户洞察、指标设计、优先级取舍、跨团队推动。
- 测试工程师：质量策略、测试覆盖、异常场景、自动化建设、上线风险。
- 研发主管：技术决策、架构权衡、团队培养、工程质量、复杂问题解决。

难点：

- 不同岗位对同一项目经历的评价重点不同。
- 同一候选人在不同岗位下的潜力判断可能不同。
- 通用问题库很难覆盖岗位里的真实工作场景。

风险：

- 如果一开始做太多岗位，每个岗位都会很浅。
- 如果岗位模板过于固定，又会失去对不同公司场景的适配能力。

缓解方式：

- MVP 第一版先选产品经理岗位做深，验证通过后再扩展第二、第三个岗位。
- 每个岗位沉淀独立的能力矩阵、追问策略和评分权重。
- 让 HR 和业务面试官参与校准岗位模板。
- 对每个岗位持续收集“系统判断 vs 人工判断”的差异，用于迭代追问模型。

#### 9.3.3 多轮追问的上下文控制

交互式面试不是一次性生成问题，而是根据回答继续追问。

难点：

- 模型需要记住候选人前面的回答。
- 追问要围绕关键疑点，不应被候选人带偏。
- 追问要控制深度，不能无限追问。
- 不同面试官之间要共享已验证信息。

风险：

- 面试过程变成长聊天，缺乏结构。
- 模型重复提问。
- 候选人回答含糊时，模型没有有效追问。

缓解方式：

- 维护面试状态：已问问题、已验证点、未验证点、风险点。
- 每轮追问前选择最重要的一个验证目标。
- 对每个问题设置最大追问次数。

#### 9.3.4 评分稳定性

招聘场景要求评估结果稳定，不能同一份简历今天 85 分、明天 62 分。

难点：

- LLM 输出天然存在波动。
- 开放式回答很难统一量化。
- 不同岗位的评分权重不同。

风险：

- 招聘团队不信任系统。
- 候选人横向比较失去意义。
- 系统难以复盘。

缓解方式：

- 先生成结构化事实，再基于规则评分。
- 将 LLM 用于分析和解释，将关键评分规则显式化。
- 对评分设置档位描述，而不是只输出单一数字。
- 保存每次评估的模型版本、Prompt 版本和评分规则版本。

### 9.4 数据与合规风险

#### 9.4.1 简历属于敏感个人信息

简历可能包含：

- 姓名、手机号、邮箱。
- 学历和工作经历。
- 薪资期望。
- 证件、照片、年龄、婚育等敏感信息。

风险：

- 数据泄露。
- 未授权使用候选人信息。
- 评估结果被不当传播。
- 不符合企业内部数据合规要求。

缓解方式：

- 最小化存储，默认只保存必要字段。
- 对联系方式等字段脱敏。
- 支持本地部署或私有化部署。
- 明确数据保留周期和删除机制。
- 限制报告分享权限。

#### 9.4.2 招聘公平性与偏见

系统可能受到以下因素影响：

- 学历偏见。
- 大厂背景偏见。
- 年龄或性别相关暗示。
- 表达风格偏见。
- 行业背景偏见。

风险：

- 误伤非典型优秀候选人。
- 放大招聘中的隐性偏见。
- 给企业带来合规和声誉风险。

缓解方式：

- 默认不把年龄、性别、婚育、照片等信息作为评分依据。
- 报告中标注“不可用于判断”的敏感字段。
- 评分必须基于岗位相关证据。
- 支持人工复核和申诉式复盘。

### 9.5 用户体验难点

#### 9.5.1 招聘方需要的是“可决策”，不是长报告

用户不一定想看很长的模型分析，真正需要的是：

- 这个人能不能进下一轮。
- 哪些地方必须追问。
- 哪些能力已经有较强证据。
- 哪些结论还不确定。
- 对比其他候选人时有什么优势和短板。

风险：

- 报告太长，招聘方不看。
- 报告太短，又缺少证据。
- 结论模糊，无法辅助决策。

缓解方式：

- 报告首页给出结论摘要。
- 详细证据放在后面。
- 使用“推荐、谨慎推荐、不推荐”而不是只给分数。
- 输出“下一轮必问题”。

#### 9.5.2 候选人体验要避免压迫感

虚拟面试官如果设计不好，会显得像“审讯系统”。

风险：

- 候选人紧张或反感。
- 好候选人因为体验差而流失。
- 企业雇主品牌受损。

缓解方式：

- 问题表达保持专业、具体、尊重。
- 追问聚焦事实还原，不做价值审判。
- 给候选人说明评估目的和隐私边界。
- 避免连续高压追问，控制节奏。

### 9.6 工程落地难点

#### 9.6.1 文档解析质量

简历格式可能是：

- PDF。
- Word。
- 图片扫描件。
- 在线导出的格式。
- 中英文混排。
- 表格布局。

风险：

- 文本抽取错乱。
- 项目段落被拆散。
- 表格信息丢失。
- 图片简历无法识别。

缓解方式：

- MVP 阶段先支持纯文本粘贴。
- 第二阶段支持 PDF 和 Word。
- 对解析失败的内容给出用户可编辑结果。

#### 9.6.2 成本与延迟控制

完整流程可能包含：

- 简历解析。
- 岗位解析。
- 硬性匹配。
- 面试官生成。
- 多轮追问。
- 回答评分。
- 报告生成。

风险：

- 单次评估调用成本过高。
- 用户等待时间过长。
- 多候选人批量评估时系统压力大。

缓解方式：

- 缓存岗位解析结果。
- 将长流程拆成异步任务。
- 对不同步骤使用不同成本的模型。
- 报告生成支持流式输出。

#### 9.6.3 可观测性和可复盘

招聘评估必须能复盘。

需要记录：

- 输入版本。
- 解析结果。
- Prompt 版本。
- 模型版本。
- 评分规则版本。
- 关键中间判断。
- 人工修改记录。

风险：

- 用户质疑结果时无法解释。
- 评分规则迭代后历史结果不可比较。
- 出现错误判断时无法定位原因。

缓解方式：

- 建立评估日志。
- 每份报告绑定版本号。
- 支持重新评估和差异对比。

### 9.7 最大的不确定性

这个产品最大的未知不是技术能不能生成问题，而是招聘团队是否认可这种工作流，以及真实用户是否愿意持续使用、复用并推荐这个工具。

需要验证的关键假设：

- 招聘方是否愿意先做硬性匹配，再做潜力评估。
- 招聘方是否相信虚拟面试官的问题质量。
- 面试官是否愿意使用系统生成的问题，而不是只凭经验问。
- 候选人是否能接受虚拟面试式追问。
- 最终报告是否真的能提高筛选效率和面试质量。
- 招聘方是否认可“每个结论都要有证据链”的报告方式。
- 资深面试官是否认为系统发现的亮点、风险点和追问方向有参考价值。
- HR 或招聘经理是否愿意把它作为面试准备工具持续使用。
- 用户是否愿意为完整报告、证据链或团队协作能力付费。
- 免费低门槛工具是否能带来第一批真实用户。

建议优先验证：

- 1 个真实产品经理岗位。
- 10 份真实或脱敏的产品经理简历。
- 3 位真实招聘参与者。
- 10 个产品经理岗位相关的免费工具试用用户。
- 对比人工面试前后的判断差异。
- 对比系统硬性匹配结论与资深面试官判断的差异。
- 对比系统生成追问与资深面试官实际追问的差异。
- 记录系统问题是否被面试官实际采用或改写采用。
- 记录同一用户是否重复使用，以及是否愿意推荐给其他 HR 或面试官。
- 记录每个差异背后的原因：证据遗漏、权重不同、岗位理解不同、系统误判或人工标准不一致。

如果这个验证不通过，应该优先调整工作流，而不是继续堆功能。

## 10. 推荐开发顺序

结合当前 Web MVP 的完成状态，后续推荐顺序调整为：

1. 保持第一版主场景为产品经理 / 项目型岗位的面试前准备，不急于泛岗位扩展。
2. 用 10 份左右脱敏简历和真实 JD 跑当前 Web MVP，记录硬性匹配、证据等级、追问问题和人工判断差异。
3. 建立一份人工评测表，统计硬性匹配一致率、系统追问被采用率、报告可读性、证据可信度和高频误判原因。
4. 将当前前端中的证据等级、闸口判断、Offer 杠杆和面试官视角抽成结构化对象，补强 Evidence Schema 与 EvaluationRun Schema。
5. 将人工反馈从“追加到报告正文”升级为结构化字段：同意程度、问题采用情况、证据充分性、误判原因、面试后验证结果。
6. 拆分 `apps/web/app.js`，优先拆出 i18n、报告生成、证据判断、PDF 导出、模型调用、Offer 推演和反馈处理模块。
7. 固化产品经理岗位能力矩阵和面试官视角库，把当前固定逻辑整理为第一批 SkillDefinition 示例。
8. 打磨真实模型 Prompt 稳定性，确保真实模型输出与 Mock 报告的结构、证据链、闸口判断和双语要求一致。
9. 做系统判断与人工判断的差异分析，优先修正证据误引、缺证误判、问题泛化和报告过长问题。
10. 在追问质量和复用率验证通过后，再扩展 Offer 沙盘能力，包括候选人回答模拟、面试官分歧推演、谈薪策略和下一步行动建议。
11. 最后再考虑候选人在线回答、多轮自动追问、候选人对比、团队协作、ATS 集成和企业版权限。

## 11. 第一版里程碑

第一周：

- 与 HR 和业务面试官确认产品经理作为第一版试点岗位，并明确目标职级和典型业务场景。
- 收集或准备 10 份产品经理脱敏简历、产品经理岗位 JD 和人工评价样本。
- 完成需求梳理，重点确认硬性匹配、追问生成和潜力相关行为证据的边界。
- 建立产品经理岗位能力矩阵，覆盖需求洞察、业务理解、指标意识、优先级判断、跨团队推动、用户价值判断、项目复盘和沟通表达。
- 明确第一版不做候选人在线面试、多轮自动追问、多岗位模板、ATS 集成和复杂权限系统。
- 设计 CandidateProfile、JobProfile、Evidence、EvaluationResult、EvaluationRun 数据结构。
- 预留 SkillDefinition 和 OfferSimulationRun 数据结构，明确后续“Offer 沙盘 + 面试官 Skill 库”的扩展方向。
- 写出 5 个核心 Prompt，并要求输出 evidence id。
- 写出第一版产品经理岗位 Skill、HR Skill 和业务面试官 Skill 的定义草案。

第二周：

- 完成简历解析、岗位解析和硬性匹配。
- 为每条匹配点、差距点和风险点绑定证据来源。
- 完成第一版证据链展示格式。
- 完成 EvaluationRun 保存逻辑，确保每次评估都可复盘。
- 输出 Markdown 格式面试准备报告。
- 报告首页突出最大亮点、最大风险和 3 到 5 个必问问题。
- 与资深面试官对照 10 份样本的硬性匹配结论，记录差异原因。

第三周：

- 针对产品经理岗位加入垂直化追问问题生成。
- 为每个问题标注对应能力维度、简历证据、JD 要求、优秀回答特征和风险回答特征。
- 生成潜力相关行为证据与待验证假设，但不输出确定性潜力评分。
- 用内置 Skill 约束追问生成，确保问题符合产品经理岗位、HR 和业务面试官的不同关注点。
- 加入人工反馈表，记录同意、部分同意、不同意、需要补充证据、问题是否实际采用。
- 与资深面试官对照系统追问质量，记录高频分歧和问题被采用率。

第四周：

- 做一个简单 Web 页面。
- 支持导入简历和 JD。
- 支持生成面试准备报告。
- 支持查看两步走证据链。
- 支持 HR 或业务面试官标注“同意、部分同意、不同意、需要补充证据”。
- 支持标注系统问题是否被实际采用或改写采用。
- 将最小 Web MVP 部署到 GitHub Pages，用于演示输入流程、证据链、追问清单和报告样例。
- 第一版采用用户每次打开页面临时设置大模型 Key 的方式，不保存用户信息和 Key。
- 报告生成后支持下载到本地，刷新页面后清空输入、Key 和报告结果。
- 提供 `.env.example` 和配置说明，仅作为本地开发或用户自建代理的可选方案，明确真实 Key 不进入代码仓库。
- 邀请真实招聘场景试用并收集反馈，至少覆盖 3 位真实招聘参与者。
- 尝试发布一个免费低门槛的“产品经理面试问题生成器”，收集第一批真实用户。
- 记录复用率、推荐意愿和付费意愿访谈结果。
- 输出 MVP 验证结论：继续推进、调整工作流、收窄岗位范围、补强 GTM 或暂停。
- 同步评估是否进入下一阶段 Offer 沙盘：若系统问题采用率、复用率和人工反馈质量达标，再增加候选人回答模拟、谈薪策略和面试官分歧推演。
