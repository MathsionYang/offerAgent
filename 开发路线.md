# 虚拟面试官候选人潜力评估系统开发路线

## 1. 产品目标

构建一个基于“简历 + 岗位要求 + 项目角色 + 候选人回答证据”的候选人评估系统。系统先完成简历与岗位的硬性匹配，再根据候选人在项目中的真实职责，生成项目经理、研发主管、HRBP 等虚拟面试官，围绕项目经历进行多视角追问，最终输出候选人的岗位匹配度、项目真实性、能力短板与长期潜力判断。

产品定位不是做一个“黑盒 AI 面试官”，而是做一个可解释的招聘辅助决策系统。它不仅要回答“候选人是否匹配”，还必须能讲清楚“为什么匹配、为什么不匹配、为什么认为有潜力”。每一个匹配结论、风险判断和潜力判断，都应该可以追溯到简历中的具体描述、岗位要求中的具体条目，或候选人在面试回答中的具体证据。

核心原则：

- 第一步先判断“是否匹配岗位要求”，不直接谈潜力。
- 第二步基于第一步发现的亮点、疑点和差距，再评估“未来潜力”。
- 两步走必须形成证据链：硬性匹配引用简历和 JD 证据，潜力判断引用项目追问和候选人回答证据。
- 报告必须区分事实、推断和待验证项，避免把模型猜测包装成确定结论。
- 面试官问题必须围绕候选人真实项目经历展开，避免泛泛而谈。
- 虚拟面试官需要有明确角色、关注维度和追问风格。
- 优先在特定岗位上做深，例如项目经理、产品经理、测试工程师、研发主管，再逐步扩展到通用招聘场景。

## 2. 核心用户流程

1. 用户输入候选人简历。
2. 用户输入岗位要求。
3. 系统抽取岗位关键词、能力要求、项目经验要求和隐性要求。
4. 系统解析简历中的技能、项目、职责、成果和候选人角色。
5. 系统为岗位要求和简历信息建立证据索引，记录每个判断来自哪里。
6. 系统完成第一步：简历硬性匹配，并输出匹配点、差距点、待验证点及对应证据。
7. 系统识别简历中的亮点、疑点、风险点和需要追问的项目。
8. 系统根据岗位类型、项目类型和候选人角色，生成虚拟面试官团队。
9. 虚拟面试官围绕项目进行结构化提问和多轮追问。
10. 系统把候选人回答沉淀为新的证据，更新项目真实性、贡献度和能力判断。
11. 系统完成第二步：潜力匹配，并明确每个潜力判断对应的回答证据和置信度。
12. 系统输出最终评估报告，包括岗位匹配、证据链、项目可信度、能力画像、潜力等级、风险点和录用建议。

## 3. MVP 版本范围

MVP 阶段先做成一个可用的文本评估工具，不追求复杂前端。MVP 的核心不是证明系统“能生成报告”，而是验证招聘团队是否认可“两步走 + 证据链 + 多角色追问”的工作流。

MVP 需要直面三个关键不确定性：

- 系统的硬性匹配结论是否接近资深招聘方或资深面试官的判断。
- 系统生成的潜力判断是否能被具体证据支持，而不是像主观点评。
- 虚拟面试官的问题是否真的能帮助业务面试官识别候选人的真实贡献。

建议 MVP 先选择 1 到 3 个具体岗位做深，例如项目经理、产品经理、测试工程师或研发主管，不建议一开始覆盖所有岗位。

### 3.1 输入模块

- 简历文本输入。
- 岗位要求文本输入。
- 可选输入：目标职级、面试方向、公司类型、是否偏技术或管理。

### 3.2 简历解析模块

需要抽取以下信息：

- 基本背景：年限、行业、学历、公司经历。
- 技能栈：语言、框架、数据库、中间件、云服务、工具链。
- 项目经历：项目名称、业务背景、技术方案、职责、成果。
- 候选人角色：主导、负责、参与、协作、支持。
- 可量化成果：性能提升、成本下降、稳定性提升、营收影响、团队规模。

### 3.3 岗位解析模块

需要抽取以下信息：

- 必备技能。
- 加分技能。
- 核心职责。
- 项目经验要求。
- 管理或协作要求。
- 隐性要求，例如抗压、owner 意识、跨部门推动、架构能力、业务理解。

### 3.4 第一步：硬性匹配模块

输出内容：

- 总体匹配结论：高度匹配、基本匹配、部分匹配、差距较大。
- 技能匹配表。
- 项目经验匹配表。
- 职责匹配表。
- 明显亮点。
- 明显缺口。
- 简历疑点。
- 每条匹配结论对应的证据来源。
- 每条不确定结论对应的验证问题。

关键规则：

- 只基于简历和岗位要求做判断。
- 不因为候选人表达好就提前判断潜力。
- 对“参与、负责、主导”等词做贡献度拆解。
- 对没有证据支撑的成果标记为待验证。
- 没有证据的结论不得进入“已匹配”，只能进入“待验证”。
- 第一阶段输出必须能回答：这条判断来自简历哪句话、岗位哪条要求。

### 3.5 第二步：潜力评估模块

基于第一步结果生成面试官团队。

第二步不是重新评价候选人，而是沿着第一步发现的亮点、差距和疑点继续验证。潜力判断只能来自候选人在项目追问中的行为证据，例如如何拆解问题、如何做取舍、如何复盘失败、如何推动他人协作。

默认虚拟面试官：

- 项目经理：关注交付、进度、资源协调、风险控制、跨部门沟通。
- 研发主管：关注技术深度、架构判断、工程质量、复杂问题解决。
- HRBP / 潜力星探：关注学习敏锐度、自驱力、复盘能力、文化契合度。

可扩展面试官：

- 架构师：关注系统设计、扩展性、稳定性、技术选型。
- 业务负责人：关注业务理解、指标意识、用户价值。
- 测试负责人：关注质量意识、异常场景、上线风险。
- 数据负责人：关注数据口径、指标归因、实验设计。

### 3.6 面试追问模块

每个面试官生成：

- 3 到 5 个核心问题。
- 每个问题配 2 到 3 个追问。
- 每个问题标注评估维度。
- 每个问题说明优秀回答特征和风险回答特征。

问题类型：

- 项目真实性追问。
- 贡献度追问。
- 决策逻辑追问。
- 冲突处理追问。
- 失败复盘追问。
- 技术权衡追问。
- 结果归因追问。

### 3.7 评估报告模块

报告结构：

- 候选人与岗位硬性匹配结论。
- 两步走证据链摘要。
- 项目经历可信度判断。
- 各虚拟面试官评价。
- 四类潜力维度评分：
  - 思维模型。
  - 变革敏锐度。
  - 人际影响力。
  - 结果驱动力。
- 风险点。
- 建议继续追问的问题。
- 与资深面试官人工判断的差异记录。
- 录用建议：强烈推荐、推荐、谨慎推进、不推荐。

## 4. 分阶段开发路线

### 阶段一：规则化原型

目标：快速验证“两步走评估流程 + 证据链解释 + 多角色追问”是否可被招聘团队接受。

功能：

- 支持粘贴简历和岗位要求。
- 用固定 prompt 完成简历解析和岗位解析。
- 输出第一步硬性匹配结果。
- 输出默认三位虚拟面试官的问题清单。
- 生成一版静态评估报告。
- 每条匹配结论必须绑定简历或 JD 原文证据。
- 每个潜力判断必须绑定候选人项目经历或模拟回答证据。
- 支持选择一个岗位模板，例如项目经理、产品经理、测试工程师或研发主管。

交付物：

- Prompt 模板。
- 基础命令行或简单 Web 页面。
- 一份完整示例报告。
- 证据链展示格式。
- 至少一个垂直岗位的追问模板。
- 与资深面试官人工判断的对照记录表。

验收标准：

- 能清晰区分“岗位匹配”和“潜力评估”。
- 能从项目经历中生成具体追问，而不是生成通用八股问题。
- 能识别简历中“主导、负责、参与”背后的待验证点。
- 每个关键结论都能追溯到证据。
- 招聘方能看懂系统为什么得出该结论。
- 与资深面试官判断不一致时，系统能暴露差异原因，而不是只给不同分数。

### 阶段二：结构化数据与评分体系

目标：让评估结果更稳定、可比较、可解释。

功能：

- 将简历解析为结构化 JSON。
- 将岗位要求解析为结构化 JSON。
- 建立技能、职责、项目、潜力四类评分模型。
- 支持评分理由引用原文证据。
- 支持按岗位类型调整权重。
- 建立 Evidence Schema，统一管理简历证据、JD 证据、回答证据和人工反馈证据。
- 建立垂直岗位评分权重，例如项目经理更重交付和协同，产品经理更重需求判断和业务指标，测试工程师更重质量风险和异常场景。
- 支持人工面试官对系统结论进行标注：同意、部分同意、不同意、需要补充证据。

交付物：

- Resume Schema。
- Job Requirement Schema。
- Evidence Schema。
- Evaluation Schema。
- 评分规则说明。
- 岗位垂直化权重配置。
- 人工评审反馈表。

验收标准：

- 同一份简历和岗位多次评估结果稳定。
- 每个评分都有证据来源。
- 缺少证据时系统明确标注“不确定”，而不是编造判断。
- 不同岗位使用不同权重和追问策略。
- 能统计系统判断与资深面试官判断的一致率和分歧点。

### 阶段三：岗位垂直化追问模型

目标：在特定岗位上做深，形成区别于通用 AI 面试官的场景适配能力。

优先岗位：

- 项目经理：重点验证项目交付、风险管理、资源协调、干系人沟通、复盘能力。
- 产品经理：重点验证需求洞察、业务目标、优先级判断、数据意识、跨团队推动。
- 测试工程师：重点验证质量策略、异常场景、自动化测试、上线风险、缺陷归因。
- 研发主管：重点验证技术决策、架构权衡、团队管理、工程质量、复杂问题解决。

功能：

- 每类岗位维护独立的能力模型。
- 每类岗位维护独立的虚拟面试官组合。
- 每类岗位维护独立的追问策略和优秀回答特征。
- 根据候选人在项目中的角色选择追问深度。
- 对岗位内常见包装话术建立验证问题库。

交付物：

- 项目经理岗位模板。
- 产品经理岗位模板。
- 测试工程师岗位模板。
- 研发主管岗位模板。
- 垂直岗位问题库。
- 岗位能力矩阵。

验收标准：

- 同一候选人在不同岗位下会得到不同追问，而不是通用问题复用。
- 业务面试官认为问题能覆盖该岗位的关键风险。
- 对项目角色的追问能区分负责人、核心参与者和普通参与者。

### 阶段四：交互式虚拟面试

目标：从静态问题清单升级为可对话面试。

功能：

- 候选人逐题回答。
- 虚拟面试官根据回答自动追问。
- 不同面试官保留自己的关注维度和风格。
- 系统根据回答实时更新风险点和潜力判断。
- 支持中途切换面试官。

交付物：

- 面试会话状态管理。
- 面试官角色配置。
- 追问策略。
- 回答评分器。

验收标准：

- 追问能引用候选人刚才的回答。
- 面试官不会偏离候选人的项目经历。
- 能把“回答流畅”与“真实贡献”区分开。
- 回答产生的新证据能自动进入证据链。
- 第二步潜力评估能解释它引用了哪些回答。

### 阶段五：报告与招聘决策辅助

目标：服务真实招聘决策。

功能：

- 生成标准化面试报告。
- 生成面试官纪要。
- 输出候选人风险雷达图或能力矩阵。
- 输出下一轮面试建议。
- 支持同岗位多个候选人横向对比。
- 输出“系统判断 vs 人工判断”差异分析。
- 输出每个结论的证据链和置信度。
- 支持招聘方对报告结论进行反馈，形成后续优化数据。

交付物：

- PDF / Markdown 报告。
- 候选人对比视图。
- 面试结论模板。
- 证据链详情页。
- 人工反馈与差异分析页。

验收标准：

- 报告能直接给招聘团队使用。
- 能明确说明推荐或不推荐的依据。
- 能指出下一轮面试需要验证的关键问题。
- 招聘团队能通过报告复盘每个关键判断。
- 报告能支持“辅助决策”，而不是只做“辅助筛选”。

### 阶段六：产品化与团队协作

目标：让系统可在招聘流程中持续使用。

功能：

- 用户登录和权限管理。
- 岗位库。
- 候选人库。
- 面试记录归档。
- 评估标准版本管理。
- 团队协作评论。
- 与 ATS 或招聘系统集成。

交付物：

- Web 应用。
- 数据库设计。
- API 接口。
- 权限模型。

验收标准：

- 支持真实招聘团队多人使用。
- 历史面试记录可追溯。
- 岗位标准和评估结果可复盘。

## 5. 推荐技术架构

### 5.1 前端

- React / Next.js。
- 简历与岗位输入页面。
- 面试官配置页面。
- 交互式面试页面。
- 评估报告页面。

### 5.2 后端

- Python FastAPI 或 Node.js NestJS。
- LLM 调用服务。
- 文档解析服务。
- 评分与报告生成服务。
- 面试会话状态服务。

### 5.3 数据库

- PostgreSQL：存储岗位、候选人、评估报告、面试记录。
- Redis：存储临时会话和异步任务状态。
- 对象存储：保存原始简历文件和报告文件。

### 5.4 LLM 能力层

- 简历结构化抽取。
- 岗位要求结构化抽取。
- 匹配度分析。
- 虚拟面试官生成。
- 追问生成。
- 回答评分。
- 报告生成。

## 6. Prompt 模板规划

### 6.1 简历解析 Prompt

目标：把非结构化简历转为结构化候选人画像。

输出字段：

- 候选人基本信息。
- 技能标签。
- 项目列表。
- 每个项目中的角色判断。
- 每个项目的可验证成果。
- 简历疑点。

### 6.2 岗位解析 Prompt

目标：把岗位 JD 转为结构化要求。

输出字段：

- 必备条件。
- 加分条件。
- 核心职责。
- 技术栈。
- 项目经验。
- 管理协作要求。
- 隐性能力要求。

### 6.3 硬性匹配 Prompt

目标：先评估候选人是否满足岗位要求。

约束：

- 不评估潜力。
- 不根据想象补全经历。
- 每条判断必须引用简历或岗位原文依据。

### 6.4 虚拟面试官生成 Prompt

目标：根据候选人项目角色和岗位要求生成合适的面试官。

输出字段：

- 面试官姓名。
- 面试官角色。
- 核心关注维度。
- 提问风格。
- 需要验证的问题。

### 6.5 潜力评估 Prompt

目标：基于候选人回答评估长期发展上限。

评估维度：

- 思维模型。
- 变革敏锐度。
- 人际影响力。
- 结果驱动力。
- 学习敏锐度。
- Owner 意识。

## 7. 数据结构草案

### 7.1 CandidateProfile

```json
{
  "basic_info": {},
  "skills": [],
  "projects": [
    {
      "name": "",
      "background": "",
      "role_claim": "",
      "responsibilities": [],
      "tech_stack": [],
      "results": [],
      "source_evidence_ids": [],
      "evidence_level": "high | medium | low",
      "questions_to_verify": []
    }
  ],
  "risks": [],
  "highlights": []
}
```

### 7.2 JobProfile

```json
{
  "title": "",
  "level": "",
  "must_have": [],
  "nice_to_have": [],
  "responsibilities": [],
  "project_requirements": [],
  "soft_requirements": [],
  "weights": {}
}
```

### 7.3 Evidence

```json
{
  "id": "",
  "source_type": "resume | job_requirement | candidate_answer | interviewer_feedback | system_inference",
  "source_ref": "",
  "original_text": "",
  "normalized_claim": "",
  "related_dimension": "",
  "confidence": 0,
  "created_at": ""
}
```

说明：

- resume：来自简历原文，例如某段项目描述。

- system_inference：系统基于多个证据做出的推断，必须引用其他 evidence id。

## 8. 风险与注意事项

- 不要把虚拟面试官做成只会问通用问题的聊天机器人。
- 不要在第一步就因为候选人“看起来聪明”而高估潜力。
- 不要把简历中的“参与”自动等同于“主导”。
- 不要编造候选人没有提供的经历。
- 对模型不确定的判断必须显式标注。
- 评估报告应作为招聘辅助，不应替代最终人工判断。

## 9. 需求深挖：难点与风险点

这个系统的难点不在于“生成几个面试问题”，而在于能否把简历、岗位要求、项目角色、候选人回答和最终判断之间的证据链建立起来。真正有价值的评估系统必须做到：先判断岗位匹配，再围绕候选人实际项目追问，最后谨慎推断潜力。

### 9.1 业务本质难点

#### 9.1.1 岗位要求经常是不完整或失真的

很多 JD 并不能真实表达岗位需求。常见情况包括：

- JD 写了很多技术栈，但真正关键的是业务复杂度和交付经验。
- JD 写的是高级工程师，实际希望招的是能带项目的小组负责人。
- JD 强调“沟通能力”，但没有说明是跨部门推动、客户沟通，还是团队协作。
- JD 把必备项和加分项混在一起，导致系统难以判断权重。
- 不同公司对同一个职级的能力要求差异很大。

产品难点：

- 需要把 JD 拆成“显性要求”和“隐性要求”。
- 需要允许用户确认或调整岗位权重。
- 不能只做关键词匹配，否则会高估简历包装能力强的人。

风险：

- 如果岗位解析错了，后续匹配和面试问题都会偏。
- 如果系统不能解释“为什么这样理解岗位”，招聘方很难信任结果。

缓解方式：

- 在岗位解析后增加确认页，让用户确认必备项、加分项和权重。
- 对不清晰的岗位要求标注“需要人工确认”。
- 支持岗位模板，例如后端工程师、项目经理、研发主管、数据分析师等。

#### 9.1.2 简历信息天然带有包装和噪音

候选人简历常见问题：

- 用“负责”包装“参与”。
- 用团队成果包装个人贡献。
- 只写技术名词，不写决策过程。
- 只写项目结果，不写上下文和约束。
- 项目经历缺少失败、冲突、取舍和复盘。
- 成果数据没有口径，例如“性能提升 50%”不知道提升的是接口耗时、吞吐量还是页面加载速度。

产品难点：

- 系统不能把简历描述直接当事实。
- 需要判断“证据强度”，而不是只判断“是否出现关键词”。
- 需要区分“候选人声称的角色”和“从描述中推断的真实角色”。

风险：

- 过度相信简历会导致误判。
- 过度怀疑简历会导致误伤优秀候选人。
- 如果问题太像审问，候选人体验会变差。

缓解方式：

- 每个项目增加证据等级：高、中、低。
- 对“主导、负责、参与、协作”建立贡献度验证规则。
- 面试问题采用“请还原当时场景”的方式，而不是直接质疑。

#### 9.1.3 “潜力”很难被直接测量

潜力不是简历上的事实，而是基于行为证据的推断。系统需要避免把以下因素误判为潜力：

- 表达能力强。
- 熟悉面试套路。
- 简历写得漂亮。
- 名校或大厂背景。
- 对流行概念掌握较多。

真正更接近潜力的证据包括：

- 面对未知问题时是否能结构化拆解。
- 遇到变化时是否主动调整策略。
- 是否能从失败中提炼可复用经验。
- 是否能在资源不足时推动结果。
- 是否能把业务目标、技术方案和组织协作连起来。

产品难点：

- 潜力评估必须建立在具体项目行为上。
- 不能只通过一次回答给出武断结论。
- 需要区分“当前能力不足”和“成长速度很快”。

风险：

- 潜力评估容易主观化。
- 模型可能受到候选人表达风格影响。
- 如果报告写得太绝对，会影响招聘公平性。

缓解方式：

- 潜力评分必须引用候选人的回答证据。
- 使用“倾向判断”而不是绝对判断。
- 给出置信度，例如高置信、中置信、低置信。

### 9.2 核心产品难点

#### 9.2.1 两步评估顺序必须被产品强制保证

这个系统的关键设计是：

1. 第一步先做硬性匹配。
2. 第二步再做潜力匹配。

难点在于模型很容易提前进入“综合评价”，把匹配和潜力混在一起。例如：

- 简历硬条件不够，但候选人看起来有潜力，系统提前给高推荐。
- 简历硬条件很好，但项目贡献度不清楚，系统直接推荐。
- 岗位要求是高级研发主管，候选人只做过执行型开发，系统因为技术栈匹配而忽略管理差距。

产品风险：

- 招聘方会误以为这是一个“智能打分器”，忽略证据边界。
- 候选人可能因为潜力描述被过度拔高或低估。

设计要求：

- 第一阶段报告中禁止出现潜力评分。
- 第二阶段必须引用第一阶段的亮点、差距和疑点。
- 最终报告要分开呈现“当前匹配度”和“未来潜力”。

#### 9.2.2 虚拟面试官不能只是换语气

“项目经理、研发主管、HRBP”不能只是名字不同，问题必须真的不同。

项目经理应该追问：

- 目标怎么定的。
- 进度如何拆解。
- 风险如何提前发现。
- 资源冲突怎么处理。
- 交付失败时怎么补救。

研发主管应该追问：

- 技术方案为什么这样选。
- 替代方案是什么。
- 架构边界在哪里。
- 性能、稳定性、可维护性如何权衡。
- 代码质量和工程规范如何保证。

HRBP / 潜力星探应该追问：

- 为什么选择承担这个角色。
- 面对压力如何调整。
- 从失败中学到了什么。
- 如何接受反馈。
- 长期职业驱动力是什么。

产品难点：

- 面试官角色需要配置化，而不是硬编码。
- 面试官问题要基于候选人的项目和岗位要求生成。
- 不同面试官之间需要互补，避免重复问同一类问题。

风险：

- 如果问题重复，用户会觉得系统只是套模板。
- 如果问题太发散，会偏离岗位评估目标。
- 如果问题太尖锐，可能造成不友好的候选人体验。

缓解方式：

- 为每个面试官定义关注维度、问题边界和禁问范围。
- 生成问题后做去重和覆盖度检查。
- 每个问题都绑定一个评估目标。

#### 9.2.3 项目角色识别是评估质量的核心

用户明确提到“根据面试者在项目中承担的角色”，这说明系统必须围绕候选人的真实项目角色设计面试。

项目角色可能包括：

- 项目负责人。
- 技术负责人。
- 核心开发。
- 模块开发。
- 协作支持。
- 需求分析。
- 测试或交付支持。
- 跨部门推动者。

难点：

- 简历中的角色描述往往模糊。
- 同一个项目中候选人可能有多个角色。
- 不同角色对应不同的评估重点。

例子：

- 如果候选人是项目负责人，应重点追问目标拆解、风险控制、资源协调和结果复盘。
- 如果候选人是技术负责人，应重点追问技术选型、架构权衡、性能稳定性和工程质量。
- 如果候选人是模块开发，应重点追问边界理解、实现细节、异常处理和协作方式。
- 如果候选人只是参与者，不应直接用负责人标准评估管理能力。

风险：

- 角色识别错了，问题会问偏。
- 用高级角色标准评估参与者，会不公平。
- 用低阶角色标准评估负责人，会低估候选人。

缓解方式：

- 对每个项目输出“角色声明”和“角色推断”。
- 对角色置信度低的项目优先追问确认。
- 面试官问题按角色分层生成。

### 9.3 模型与算法难点

#### 9.3.1 证据链构建

系统每个判断都应该能回答：

- 依据来自简历哪一句。
- 依据来自岗位要求哪一条。
- 依据来自候选人哪次回答。
- 这个判断的置信度是多少。
- 这是事实、推断，还是待验证假设。
- 如果与人工面试官判断不同，差异来自证据不足、权重不同，还是解释不同。

难点：

- 简历和 JD 是非结构化文本。
- 同一能力可能分散在多个项目中。
- 候选人回答可能补充、推翻或弱化简历信息。
- 潜力判断往往不是单条证据可以支撑，而是多个行为证据的组合推断。

风险：

- 没有证据链，报告会像主观点评。
- 模型可能产生听起来合理但没有依据的结论。
- 证据链如果展示得太复杂，招聘方会看不懂。
- 如果只展示结论不展示证据，产品会退化成黑盒打分器。

缓解方式：

- 使用结构化中间结果。
- 每个评分字段都要求 evidence。
- 报告中区分“事实、推断、待验证”。
- 为每个关键结论展示 1 到 3 条最关键证据，而不是把所有证据堆给用户。
- 对没有证据的内容只允许进入“待验证问题”，不允许进入最终判断。

#### 9.3.2 岗位垂直化能力模型

通用 AI 面试官容易问出“看似合理但没有岗位针对性”的问题。真正形成壁垒的方向，是在特定岗位上做深，让系统理解该岗位的真实评价逻辑。

优先建议做深的岗位：

- 项目经理：交付目标、风险控制、资源协调、干系人管理、项目复盘。
- 产品经理：需求判断、用户洞察、指标设计、优先级取舍、跨团队推动。
- 测试工程师：质量策略、测试覆盖、异常场景、自动化建设、上线风险。
- 研发主管：技术决策、架构权衡、团队培养、工程质量、复杂问题解决。

难点：

- 不同岗位对同一项目经历的评价重点不同。
- 同一候选人在不同岗位下的潜力判断可能不同。
- 通用问题库很难覆盖岗位里的真实工作场景。

风险：

- 如果一开始做太多岗位，每个岗位都会很浅。
- 如果岗位模板过于固定，又会失去对不同公司场景的适配能力。

缓解方式：

- MVP 阶段先选 1 到 3 个岗位做深。
- 每个岗位沉淀独立的能力矩阵、追问策略和评分权重。
- 让 HR 和业务面试官参与校准岗位模板。
- 对每个岗位持续收集“系统判断 vs 人工判断”的差异，用于迭代追问模型。

#### 9.3.3 多轮追问的上下文控制

交互式面试不是一次性生成问题，而是根据回答继续追问。

难点：

- 模型需要记住候选人前面的回答。
- 追问要围绕关键疑点，不应被候选人带偏。
- 追问要控制深度，不能无限追问。
- 不同面试官之间要共享已验证信息。

风险：

- 面试过程变成长聊天，缺乏结构。
- 模型重复提问。
- 候选人回答含糊时，模型没有有效追问。

缓解方式：

- 维护面试状态：已问问题、已验证点、未验证点、风险点。
- 每轮追问前选择最重要的一个验证目标。
- 对每个问题设置最大追问次数。

#### 9.3.4 评分稳定性

招聘场景要求评估结果稳定，不能同一份简历今天 85 分、明天 62 分。

难点：

- LLM 输出天然存在波动。
- 开放式回答很难统一量化。
- 不同岗位的评分权重不同。

风险：

- 招聘团队不信任系统。
- 候选人横向比较失去意义。
- 系统难以复盘。

缓解方式：

- 先生成结构化事实，再基于规则评分。
- 将 LLM 用于分析和解释，将关键评分规则显式化。
- 对评分设置档位描述，而不是只输出单一数字。
- 保存每次评估的模型版本、Prompt 版本和评分规则版本。

### 9.4 数据与合规风险

#### 9.4.1 简历属于敏感个人信息

简历可能包含：

- 姓名、手机号、邮箱。
- 学历和工作经历。
- 薪资期望。
- 证件、照片、年龄、婚育等敏感信息。

风险：

- 数据泄露。
- 未授权使用候选人信息。
- 评估结果被不当传播。
- 不符合企业内部数据合规要求。

缓解方式：

- 最小化存储，默认只保存必要字段。
- 对联系方式等字段脱敏。
- 支持本地部署或私有化部署。
- 明确数据保留周期和删除机制。
- 限制报告分享权限。

#### 9.4.2 招聘公平性与偏见

系统可能受到以下因素影响：

- 学历偏见。
- 大厂背景偏见。
- 年龄或性别相关暗示。
- 表达风格偏见。
- 行业背景偏见。

风险：

- 误伤非典型优秀候选人。
- 放大招聘中的隐性偏见。
- 给企业带来合规和声誉风险。

缓解方式：

- 默认不把年龄、性别、婚育、照片等信息作为评分依据。
- 报告中标注“不可用于判断”的敏感字段。
- 评分必须基于岗位相关证据。
- 支持人工复核和申诉式复盘。

### 9.5 用户体验难点

#### 9.5.1 招聘方需要的是“可决策”，不是长报告

用户不一定想看很长的模型分析，真正需要的是：

- 这个人能不能进下一轮。
- 哪些地方必须追问。
- 哪些能力已经有较强证据。
- 哪些结论还不确定。
- 对比其他候选人时有什么优势和短板。

风险：

- 报告太长，招聘方不看。
- 报告太短，又缺少证据。
- 结论模糊，无法辅助决策。

缓解方式：

- 报告首页给出结论摘要。
- 详细证据放在后面。
- 使用“推荐、谨慎推荐、不推荐”而不是只给分数。
- 输出“下一轮必问题”。

#### 9.5.2 候选人体验要避免压迫感

虚拟面试官如果设计不好，会显得像“审讯系统”。

风险：

- 候选人紧张或反感。
- 好候选人因为体验差而流失。
- 企业雇主品牌受损。

缓解方式：

- 问题表达保持专业、具体、尊重。
- 追问聚焦事实还原，不做价值审判。
- 给候选人说明评估目的和隐私边界。
- 避免连续高压追问，控制节奏。

### 9.6 工程落地难点

#### 9.6.1 文档解析质量

简历格式可能是：

- PDF。
- Word。
- 图片扫描件。
- 在线导出的格式。
- 中英文混排。
- 表格布局。

风险：

- 文本抽取错乱。
- 项目段落被拆散。
- 表格信息丢失。
- 图片简历无法识别。

缓解方式：

- MVP 阶段先支持纯文本粘贴。
- 第二阶段支持 PDF 和 Word。
- 对解析失败的内容给出用户可编辑结果。

#### 9.6.2 成本与延迟控制

完整流程可能包含：

- 简历解析。
- 岗位解析。
- 硬性匹配。
- 面试官生成。
- 多轮追问。
- 回答评分。
- 报告生成。

风险：

- 单次评估调用成本过高。
- 用户等待时间过长。
- 多候选人批量评估时系统压力大。

缓解方式：

- 缓存岗位解析结果。
- 将长流程拆成异步任务。
- 对不同步骤使用不同成本的模型。
- 报告生成支持流式输出。

#### 9.6.3 可观测性和可复盘

招聘评估必须能复盘。

需要记录：

- 输入版本。
- 解析结果。
- Prompt 版本。
- 模型版本。
- 评分规则版本。
- 关键中间判断。
- 人工修改记录。

风险：

- 用户质疑结果时无法解释。
- 评分规则迭代后历史结果不可比较。
- 出现错误判断时无法定位原因。

缓解方式：

- 建立评估日志。
- 每份报告绑定版本号。
- 支持重新评估和差异对比。

### 9.7 最大的不确定性

这个产品最大的未知不是技术能不能生成问题，而是招聘团队是否认可这种工作流。

需要验证的关键假设：

- 招聘方是否愿意先做硬性匹配，再做潜力评估。
- 招聘方是否相信虚拟面试官的问题质量。
- 面试官是否愿意使用系统生成的问题，而不是只凭经验问。
- 候选人是否能接受虚拟面试式追问。
- 最终报告是否真的能提高筛选效率和面试质量。
- 招聘方是否认可“每个结论都要有证据链”的报告方式。
- 资深面试官是否认为系统发现的亮点、风险点和追问方向有参考价值。

建议优先验证：

- 3 个真实岗位。
- 10 份真实或脱敏简历。
- 3 位真实招聘参与者。
- 对比人工面试前后的判断差异。
- 对比系统硬性匹配结论与资深面试官判断的差异。
- 对比系统潜力评估结论与资深面试官判断的差异。
- 记录每个差异背后的原因：证据遗漏、权重不同、岗位理解不同、系统误判或人工标准不一致。

如果这个验证不通过，应该优先调整工作流，而不是继续堆功能。

## 10. 推荐开发顺序

1. 先选定 1 到 3 个试点岗位，不做泛岗位覆盖。
2. 收集 10 份左右脱敏简历、岗位 JD 和资深面试官历史反馈。
3. 固化“两步走”输入输出格式，明确硬性匹配和潜力评估不得混在一起。
4. 建立 Evidence Schema，让每个匹配、差距、风险和潜力判断都能追溯证据。
5. 打磨第一步硬性匹配的准确性，优先对齐资深面试官判断。
6. 针对试点岗位生成垂直化虚拟面试官和追问问题。
7. 再加入第二步潜力评估，并要求每个潜力判断引用候选人回答证据。
8. 做系统判断与人工判断的差异分析，优先修正高频误判。
9. 最后做报告、候选人对比、团队协作和系统集成。

## 11. 第一版里程碑

第一周：

- 与 HR 和业务面试官确认 1 到 3 个试点岗位。
- 收集或准备 10 份脱敏简历、岗位 JD 和人工评价样本。
- 完成需求梳理，重点确认硬性匹配和潜力评估的边界。
- 设计 CandidateProfile、JobProfile、Evidence、EvaluationResult 数据结构。
- 写出 5 个核心 Prompt，并要求输出 evidence id。

第二周：

- 完成简历解析、岗位解析和硬性匹配。
- 为每条匹配点、差距点和风险点绑定证据来源。
- 完成第一版证据链展示格式。
- 输出 Markdown 格式评估报告。
- 与资深面试官对照 10 份样本的硬性匹配结论，记录差异原因。

第三周：

- 针对试点岗位加入垂直化虚拟面试官生成。
- 加入项目追问问题生成。
- 完成第一版潜力评估。
- 为每个潜力判断绑定项目证据或候选人回答证据。
- 与资深面试官对照潜力判断，记录高频分歧。

第四周：

- 做一个简单 Web 页面。
- 支持导入简历和 JD。
- 支持生成完整评估报告。
- 支持查看两步走证据链。
- 支持 HR 或业务面试官标注“同意、部分同意、不同意、需要补充证据”。
- 邀请真实招聘场景试用并收集反馈。
- 输出 MVP 验证结论：继续推进、调整工作流、收窄岗位范围或暂停。
