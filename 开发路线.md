# OfferAgent 开发路线

更新时间：2026-07-11

## 1. 产品定位

OfferAgent 当前形态是“Offer 沙盘 + 面试官 Skill 库 + EvidenceGraph + 虚拟面试委员会 + 一致性模式 + 统一语言投影”的静态 Web MVP。它不是自动招聘决策系统，也不是黑盒打分器，而是面向候选人与面试官的面试准备、证据追踪、追问辅助和报告导出工具。

核心目标：

1. 帮助候选人理解 JD、补齐项目证据、准备面试追问。
2. 帮助面试官快速识别风险、设计追问、记录人工反馈。
3. 用 EvidenceGraph 让 JD、简历、证据、问题、风险、反馈、Offer 信号、Skill 和虚拟角色互相引用。
4. 用 OfferSimulationRun 把 Offer 推演从报告文本升级为可回填状态。
5. 用 SkillDefinition 和 VirtualPanel 为后续 Skill Registry 与多角色推演打基础。
6. 用一致性模式降低相同输入下报告漂移。
7. 用统一语言投影保证用户输入保持原文，系统生成内容按当前语言展示。

## 2. 当前已完成

### 2.1 Web 工作台

已完成：

1. 每次进入先选择候选人 / 面试官身份，身份不做本地持久化。
2. 顶部固定标题栏。
3. 候选人 / 面试官分段切换。
4. 工作台 / 图谱分段切换。
5. 中文 / English 投影能力保留，当前语言选择器隐藏。
6. 生成报告后立即跳转图谱页。
7. 工作台包含输入区、Offer 沙盘、面试官视角库和默认折叠的高级模型配置。
8. 候选人简历与岗位 JD 左右布局。
9. 输入就绪度提示：简历 / JD 字数、内容不足提示和面试官角色选择状态。
10. 图谱页展示证据关系图谱、报告预览、虚拟委员会和结果摘要。
11. 图谱页以聊天气泡流式展示虚拟面试委员会讨论。
12. 首次使用页增加三步说明：输入简历 / JD、生成摘要 / 图谱 / 追问、本机缓存可清理。
13. 工作台提供三条脱敏样例路线：候选人准备、面试官评估、Offer 谈判。
14. 工作台显示本机运行缓存和反馈历史数量，并提供“清理本机缓存”入口。

3. 面试官模式显示人工反馈。
4. 候选人模式只显示候选人报告导出。

### 2.8 SkillDefinition 与虚拟面试委员会

已完成 5 个 SkillDefinition 示例：

1. HR 面试官。
2. 业务负责人。
3. 项目推进面试官。
4. 谈薪顾问。
5. 决策层压力官。

已完成轻量 MiroFish 化结构：

1. `VirtualPanel`：从岗位、JD、简历和 Skill 生成虚拟面试官角色。
2. `PanelDiscussionRound`：生成多轮轻量讨论。

4. EvidenceGraph 中可审计虚拟角色贡献。
5. 图谱页已支持虚拟委员会聊天气泡流式展示，并在生成完成后保留讨论历史。
6. 支持按讨论轮次、虚拟角色和证据节点筛选聊天记录。
7. challenges 可提升对应追问优先级。
8. 人工反馈可反向校准下一轮虚拟角色权重。

### 2.10 统一语言投影

## 3. 当前边界

1. 真实样本评测暂时不做。
2. 还没有用户账号、团队协作和云端报告保存。
3. 还没有 ATS / HRIS 集成。
4. Skill Registry 还不是完整运行时技能系统。
5. EvidenceGraph 还不是持久化知识图谱。
6. OfferSimulationRun 还没有跨会话云端保存。
7. 虚拟面试委员会仍是轻量规则驱动，不是完整多 Agent 引擎。
8. API Key 只在当前页面内存使用，不写入 localStorage；基础报告缓存和反馈历史会写入本机浏览器 localStorage。
9. 前端核心模块已完成拆分；当前 `apps/web/app.js` 约 2501 行，`apps/web/src` 包含 21 个 JS 模块，主入口主要保留页面编排、事件绑定、状态流转、模型提示词和 Mock 报告生成。
10. 视觉回归脚本已可生成截图和比较 baseline，但 baseline 更新流程还没有制度化。

## 4. 下一步优先级

### P0：稳定当前产品体验

1. 建立视觉回归 baseline 更新流程：确认截图、批准变化、更新 baseline、保留 current artifact。
2. 把当前 `smoke_test.py` 拆成 UI / schema / report / cache 四类检查。
3. 增加真实模型错误态浏览器级回归：CORS、401、402/额度、403、404/模型不存在、400/请求过长、超时、非 JSON 返回、流式中断。
4. 继续检查移动端布局，尤其是图谱过滤器、虚拟委员会筛选、摘要复制按钮和导出按钮。

### P1：让虚拟面试委员会更可审计

1. 基础可审计能力已完成：stance、influence_weight、focus、轮次/角色/证据筛选、主持人依据追溯。
2. 人工反馈影响下一轮虚拟角色权重已完成。
3. 下一步增强每条发言的“为什么影响优先级”解释。
4. 增加按 impact 类型筛选：raise_follow_up_priority、keep_as_supporting_evidence、feed_offer_simulation 等。

### P2：让图谱成为决策工具

1. 基础决策能力已完成：关键词搜索、类型筛选、风险等级、证据等级、来源过滤、高风险决策视图、反馈 diff。
2. 高风险决策视图解释文案已完成。
3. 增加过滤条件组合的可分享状态或可复制摘要。
4. 优化大图谱下的节点密度、排序和折叠。

### P3：把反馈闭环做实

1. 结构化反馈、反馈历史保存、Skill 更新建议和冲突裁决已完成。
2. 反馈影响虚拟角色 `influence_weight` 已完成。
3. 让反馈历史进一步影响下一轮问题生成和风险权重。
4. 增加反馈历史查看和单条删除入口；清空入口已通过“清理本机缓存”覆盖。
5. 为反馈历史增加导出能力，便于面试官复盘。

### P4：模块化工程结构

已完成模块化拆分。后续继续瘦身 `app.js`：

1. `prompt-builder.js`：系统提示词和用户提示词。
2. `mock-runner.js`：Mock 报告生成、分块输出和样例运行。
3. `app-state.js`：当前 run、语言、视图、受众状态。
4. `ui-bindings.js`：DOM 事件绑定。
5. `cache-controls.js`：本机缓存和反馈历史清理入口。

### P5：真实验证

当前暂不做真实样本评测。后续如恢复，应优先验证：

1. 面试官是否愿意采用问题。
2. 报告是否能减少面试准备时间。
3. 证据图谱是否帮助定位风险。
4. 人工反馈是否能沉淀成可复用规则。
5. 一致性模式是否显著降低同输入报告漂移。
6. 真实模型输出是否稳定遵守结构化运行契约。
