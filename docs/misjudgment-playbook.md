# OfferAgent 误判归因与修正手册

这份手册用于把人工反馈转化为可执行的产品、Prompt、证据和评测改进。每一次“系统判断与人工判断不一致”都不应只作为备注保存，而要归因到明确类型，并对应到下一步修正动作。

## 1. 归因原则

- 先判断是否是证据问题，再判断是否是能力权重问题。
- 先修正结构化事实，再修正报告表达。
- 不把所有不一致都归为系统错误；人工判断也可能来自岗位偏好、业务上下文或未提供材料。
- 每个误判至少绑定一个能力维度、一个证据项和一个修正动作。

## 2. 标准误判类型

| 类型 | 定义 | 典型表现 | 优先修正位置 |
| --- | --- | --- | --- |
| 证据不足 | 简历 / JD 中没有足够材料支撑结论 | 系统给出推进建议，但证据只是职责词或团队成果 | Evidence 抽取、闸口规则 |
| 证据误引 | 引用的句子与结论不匹配 | 用工具关键词支撑业务判断，用团队结果支撑个人贡献 | Evidence 匹配、Prompt 约束 |
| 权重不同 | 人工更看重某些能力，系统权重偏低或偏高 | 业务面试官更看重行业经验，系统更看重产品通用能力 | 岗位能力矩阵、评分权重 |
| 岗位理解不同 | JD 的真实优先级与文本表述不一致 | JD 写产品规划，实际要客户方案 / 交付型 PM | JD 解析、公司上下文输入 |
| 系统过度推断 | 系统把弱证据写成强结论 | 候选人“参与”被写成“主导” | 报告生成、证据等级 |
| 系统低估 | 系统没有识别相邻能力迁移价值 | B 端 SaaS 项目对垂直行业岗位有迁移性但被低估 | 迁移判断、条件性进入 |
| 问题泛化 | 追问像通用面试题，没有绑定项目细节 | “你如何做优先级判断”但没有项目锚点 | 面试官 Skill、问题生成 |
| 报告过长 | 结论有效但用户读不完 | 亮点、风险、行动项被埋在长报告后面 | 一页摘要、信息层级 |
| 合规边界不清 | 表述像自动淘汰或基于敏感字段判断 | “不建议录用”缺少人工复核提示 | 合规文案、输出边界 |

## 3. 修正动作矩阵

| 反馈字段 | 判断方式 | 修正动作 |
| --- | --- | --- |
| disagreement_reason = 证据不足 | 关键能力只有三级证据或缺证 | 降低闸口等级；把结论改成待验证；新增补证问题 |
| disagreement_reason = 权重不同 | 人工强调的能力未进入前 3 风险 | 调整岗位能力矩阵；为岗位增加权重配置 |
| disagreement_reason = 岗位理解不同 | 公司上下文改变 JD 优先级 | 增强上下文输入；让报告显式区分 JD 文本与真实业务优先级 |
| disagreement_reason = 系统误判 | 证据、权重和上下文都无法解释差异 | 修正规则或 Prompt；加入回归样本 |
| evidence_sufficiency = 不充分 | 结论缺少一到三条核心证据 | 把该结论从“已判断”改成“待验证” |
| risk_validation = 已证实 | 面试验证了系统风险 | 将问题提升为高价值追问；保留到 Skill 示例 |
| risk_validation = 已推翻 | 面试推翻了系统风险 | 降低该风险触发规则权重；补充反例 |
| question_use = 未采用 | 面试官认为问题低价值 | 判断是泛化、过长、太尖锐还是与岗位无关 |

## 4. 误判复盘流程

1. 读取当前运行内的结构化 EvaluationRun 状态、EvidenceGraph 关系和报告中的证据编号。
2. 对照人工评测表，找出 `system_gate` 与 `human_gate` 不一致样本。
3. 标记 `disagreement_reason`。
4. 找到对应 `requirement_matches`、`interview_questions` 和 `evidence`。
5. 判断问题属于证据、权重、岗位理解、问题生成、报告表达还是合规边界。
6. 写入 `recommended_fix`。
7. 将样本加入下一轮回归评测。

## 5. 处理优先级

优先修正：

- 会导致错误推进 / 错误暂缓的闸口误判。
- 证据误引。
- 面试官明确认为无法采用的问题。
- 报告首页读不出结论的问题。
- 合规边界可能被误解的问题。

可以延后：

- 只影响措辞但不影响判断的表达问题。
- 低频岗位特有问题。
- Offer 推演中的细节不足。
- 需要真实面试后数据才能判断的问题。
