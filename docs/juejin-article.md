# 我用 6000 行纯前端代码，做了一个能让候选人和面试官"看见证据"的面试评估沙盘

> 在线 Demo：https://mathsionyang.github.io/offerAgent/
> 开源地址：https://github.com/MathsionYang/offerAgent
> 适用岗位：产品经理 / 开发人员 / 技术支持 / 销售人员

## 写在前面

面试准备这件事有一个诡异的特点：**越是重要的判断，越没有"证据"。**

HR 看完一份简历，写一句"感觉不错"；业务面试官聊一个小时，凭印象打个分；候选人背了一周项目细节，结果面试官问的全是八股题。

我做这个项目的初衷很简单：**把"感觉"变成"可点击的证据节点"，把"追问清单"变成"可审计的问题库"**，让候选人和面试官不再互相猜。

它不是 ATS（Applicant Tracking System），也不是打分机器人，而是一个**纯前端、零后端、零数据库**的面试评估沙盘。今天这篇文章，我会拆解它的设计思路、核心模块和技术实现细节。

## 一、产品定位：为什么是"沙盘"而不是"评测"？

| 维度 | 传统面试辅助 | OfferAgent |
| --- | --- | --- |
| 形态 | 文档 / 表格 / 微信语音 | 浏览器内可交互工作台 |
| 结论 | 录用 / 淘汰 | **项目匹配闸口**（匹配进入 / 条件进入 / 缺证不匹配） |
| 追问 | 临时想 | **Skill 视角库 × 虚拟面试委员会**生成 |
| Offer | 谈薪经验 | **OfferSimulationRun** 三场景结构化推演 |
| 反馈 | 拍脑袋 | **FeedbackDistillation** 升级 / 降权 / 删除 / 保留 |
| 隐私 | 简历上传 | **Key 只在内存、刷新即清空** |

它不是要替代人做决策，而是**让每一个结论都能溯源、每一次追问都能审计、每一份反馈都能沉淀**。

## 二、核心设计：从"匹配闸口"开始

面试最常见的事故是：**简历上写"主导过 X 项目"，JD 上要"有 Y 经验"，结果一拍即合——直到入职才发现完全对不上。**

所以 OfferAgent 设计的第一道闸口是**"项目匹配闸口"**：

1. 从 JD 中抽取**岗位职责 + 能力要求**（标注一级 / 二级 / 三级可信度）
2. 从简历中找**项目经历锚点**（同样分可信度等级）
3. 输出匹配结果：**匹配进入 / 条件进入 / 缺证不匹配**
4. 对缺证结论**强制要求补项目材料**
5. 对表面高匹配内容**生成反包装追问**

**这一步不是黑盒打分，而是把"为什么匹配 / 为什么不匹配"摆到台面上。**

## 三、EvidenceGraph：让"证据"第一次变得可点击

这是我最满意的一个设计，也是项目的"心脏"。

传统报告的形态是：

```text
"候选人具备产品规划能力"——完了。
```

OfferAgent 把这个结论**逆向拆解成图谱节点**：

```text
job_requirement(产品规划) ← requires(0.92) ← evidence(主导 XX 项目)
                              ↘ 风险(缺少复盘机制)
                              ↘ skill(虚拟业务负责人视角)
                              ↘ feedback(未反馈)
                              ↘ offer_signal(谈判优先级提升)
```

**每一个节点都能点击跳转回报告对应段落**，让面试官能从结论反推证据，也能从证据追到结论。

节点的 9 种类型：
- `job_requirement` / `evidence` / `validation` / `risk` / `feedback`
- `offer_signal` / `skill` / `agent_persona` / `report_section`

边的关系不再是简单的"关联"，而是带 **confidence / weight / source** 的有向图：
- `supports`（支持）、`requires`（要求）、`raises`（提出）
- `risks`（风险）、`discusses`（讨论）、`challenges`（挑战）
- `reads_memory`（虚拟角色读取证据）

当图谱出现**证据缺口**时，会高亮提示"这块没简历背书，面试官需要追问验证"。

## 四、虚拟面试委员会：把"群聊式讨论"搬进浏览器

这个模块借鉴了 MiroFish 的 **"种子材料 → 图谱记忆 → Persona 生成 → 多轮模拟 → 汇总报告"** 思路，做了一个适合静态 Web 的轻量版。

### 4.1 五个内置 Skill 视角

```text
1. 虚拟 HR 面试官        —— 动机、岗位偏好、到岗约束
2. 虚拟业务负责人        —— 业务判断、指标口径、结果归因
3. 虚拟项目推进面试官    —— 里程碑、资源协调、复盘机制
4. 虚拟谈薪顾问          —— 机会选择、竞争 Offer、谈判策略
5. 决策层压力官          —— 预算剪裁、战略取舍、ROI 压力
```

### 4.2 多轮轻量讨论

```text
buildVirtualPanel()        // 从岗位/JD/简历/Skill 生成虚拟角色
buildPanelDiscussionRounds() // 围绕证据/风险/Offer 多轮讨论
buildModeratorSummary()     // 汇总共识、分歧、主导角色、最终建议
```

### 4.3 微信群聊式气泡流

报告页下方会以**聊天气泡流式**逐条展示虚拟角色的发言，生成完成后保留完整讨论记录。用户可以像看微信群一样，看着 5 个虚拟角色"吵"出一个 Offer 建议。

**这不是完整的多 Agent 仿真引擎**——它是基于规则和结构化上下文的轻量推理，但作为 MVP，已经能帮助面试官看到"不同视角如何看同一份简历"。

## 五、OfferSimulationRun：把"谈薪"从文本升级为可回填状态

以前 Offer 推演在报告里只是一段文字。OfferAgent 把它升级为**结构化状态机**：

```text
{
  scenario: "base" | "optimistic" | "conservative",
  stage: "draft" | "negotiating" | "accepted" | "rejected",
  history: [...],
  version: "v1.2.0",
  status_backfill: "可回填到下一轮问题生成、风险判断、谈判策略"
}
```

跑完三个场景的对比后，Offer 结果可以**反向回填**到：
- 下一轮追问问题生成
- 风险判断的优先级
- 虚拟谈薪顾问的策略建议

这意味着：**Offer 决策不是孤立的段落，而是影响后续整个流程的状态。**

## 六、一致性模式：解决 LLM 报告"漂移"的痛点

用 LLM 生成报告的人都知道：**同样的输入，今天生成的和昨天生成的，结论可能差 30%。**

OfferAgent 加了一致性模式来解决这个问题：

1. **输入指纹 `input_fingerprint`**：对 role + resume + jd + constraints + skills 做哈希
2. **结构化中间层 `structured_evaluation`**：先把评估结果结构化，再渲染报告
3. **本地缓存复用**：相同指纹命中缓存时直接复用基础报告
4. **真实模型请求**：`temperature: 0` + `seed`，减少随机性
5. **隐私保护**：**缓存不存 API Key，不存人工反馈**

> 缓存大小限制为 12 条，避免 localStorage 膨胀。

## 七、双模式工作台：同一个输入，两种视角

```text
候选人模式 → 隐藏人工反馈 → 只显示候选人报告
面试官模式 → 显示人工反馈 → 只显示面试官报告
```

但**输入信息不丢**——切换模式只切视图，不清空数据。这样一个候选人可以在准备面试时用"候选人模式"自检，面试官评估时切到"面试官模式"看追问库。

报告分块流式输出，逐块渲染，体验类似 ChatGPT 打字机效果。

## 八、隐私设计：把控制权完全交给用户

这是我在设计时最在意的点：

- ❌ **不需要注册或登录**
- ❌ **不上传简历到任何服务器**（除了用户主动调用的模型服务商）
- ❌ **不保存 API Key**（只在当前页面内存）
- ❌ **不保存简历、JD、报告或反馈**（刷新即清空）
- ❌ **不写 cookie / 不写后端数据库**

如果用户选择浏览器直连模型服务商，输入会发往对应服务商。**这由用户自主决定**。如果担心 CORS 限制，项目还附带了一个 **Cloudflare Worker 代理示例**：

```text
serverless/cloudflare-worker.js
```

把真实 Key 放服务端，前端通过 Worker 转发，**Key 不再暴露给浏览器**。

## 九、技术实现：纯前端 6000 行的取舍

整个 Web 端只有 4 个文件：

```text
apps/web/
├── index.html         (236 行)
├── styles.css         (3363 行)
├── app.js             (6826 行)
└── img/ (截图资源)
```

**没有 React / Vue / 任何框架**。原因很简单：

1. **可移植性**：纯 HTML + 原生 JS 可以直接丢 GitHub Pages，无需构建
2. **可读性**：所有人打开 DevTools 就能看到全部逻辑
3. **零依赖**：没有 npm install / node_modules / 供应链风险
4. **可审计**：6800 行 JS 跑一个面试评估，逻辑密度高但完全可读

**当然有代价**：

- 业务逻辑耦合在单文件（已在开发路线 P4 列入模块化计划）
- 没有虚拟 DOM，长列表性能需要手动优化
- 状态管理靠全局变量和 DOM 联动

但作为 MVP，**这种"返璞归真"反而是优势**——用户 fork 之后改一个 `app.js` 就能自定义自己的报告模板。

## 十一、部署 & 使用

### 11.1 最快 3 分钟体验

```bash
git clone https://github.com/MathsionYang/offerAgent.git
cd offerAgent/apps/web
python -m http.server 5173
# 浏览器打开 http://localhost:5173
```

默认进 **Mock Demo** 模式，**不调用任何模型**，跑完整套流程。

### 11.2 接真实模型

页面支持 6 个常用服务商下拉选择：
- OpenAI / DeepSeek / 通义千问（DashScope）/ Kimi（Moonshot）
- OpenAI-Compatible 代理 / 自定义接口（覆盖豆包、智谱、硅基流动等）

如果浏览器直连报 CORS 错，可以用项目自带的 **Cloudflare Worker 代理**：

```text
1. 部署 serverless/cloudflare-worker.js
2. 设置环境变量 UPSTREAM_BASE_URL
3. 在页面"代理 / 自定义 Base URL"填 Worker 根地址
```

### 11.3 GitHub Pages 一键部署

项目已带 `.github/workflows/pages.yml`，推 `main` 自动部署到：
```text
https://<your-name>.github.io/offerAgent/
```

## 十二、当前边界 & 后续规划

**它不是万能的**，目前明确不做：

- 用户账号 / 团队协作 / 云端报告保存
- ATS / HRIS 集成
- 真实样本评测数据集
- 完整的多 Agent 仿真引擎
- 持久化知识图谱数据库

后续优先级（来自开发路线）：

- **P0**：稳定当前体验（清理文案、Playwright 截图回归、smoke test 拆分）
- **P1**：让虚拟委员会更可审计（展示 stance、influence_weight、focus）
- **P2**：让图谱成为决策工具（节点搜索、风险等级过滤、feedback diff）
- **P3**：把反馈闭环做实（结构化反馈、SkillDefinition 更新候选）
- **P4**：模块化工程结构（i18n.js / roles.js / skills.js / graph.js / reports.js ...）

## 十三、写在最后：做这个项目的初心

我自己在求职和招人两端都待过。求职时，最怕的是**面试官问一个我简历里没写的项目细节**；招人时，最怕的是**聊得很好但入职后发现能力完全不对**。

OfferAgent 不能解决这两个问题，但它能做一件事：

> **把"感觉"和"经验"变成"可点击、可审计、可回填的结构化证据"。**

如果它能在你和候选人 / 面试官之间，**省下一次失败的面试，或多抓住一个被低估的候选人**，那这个项目就值了。
